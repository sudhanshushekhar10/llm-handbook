The Runtime Ecosystem: Brains vs. Engines

A model file is not software: it's inert numbers on disk. The runtime is the program that breathes life into them. Understanding this split explains why the same Llama weights can run on a laptop, a cloud server, or a Raspberry Pi, each via a completely different engine.

Reading time · 14 min Prereqs · Plate 13 (Quantization & GGUF) Covers · llama.cpp · Ollama · vLLM · GGUF · Triton

The core dichotomy: brain vs. engine

The biggest misconception for newcomers is thinking that "Llama 3" or "Mistral" is a piece of software you install and run. It isn't. A model is a collection of numbers: billions of floating-point weights stored in a file. On their own they do nothing, the same way a JPEG file doesn't display itself. You need a separate program (a runtime engine) to load those weights and execute the matrix multiplications that produce text.

Analogy · sheet music vs. piano

Sheet music is the model: a precise, portable description of a song (the weights). But sheet music makes no sound on its own. You need a piano (the runtime) to read those marks and turn them into actual music. A grand piano produces rich, full sound; a toy piano plays a simplified version; a digital synthesiser plays the same notes but with different characteristics. The music is identical across all three; the runtime determines how it's realised.

This split has a profound practical consequence: the same model file can be run by many different engines, each optimised for a different environment. The GGUF file for Llama 3 8B can be served via llama.cpp on a MacBook, via Ollama as a local API, or converted to run on vLLM on a cloud GPU cluster.

llama.cpp: the local hero

Meta released the original LLaMA model weights for research use. But the only way to run them was through PyTorch, which requires Python, CUDA drivers, and ideally an expensive NVIDIA GPU. It was completely inaccessible to anyone without a research-lab setup.

In March 2023, Georgi Gerganov released llama.cpp. He didn't create a new model. He wrote a completely new inference engine in C++ that reimplements all the necessary matrix maths from scratch, with no PyTorch dependency whatsoever. The result ran on a standard laptop CPU, including Apple Silicon via the Metal API.

What won, and why

llama.cpp succeeded because it was written in C++ with hand-optimised memory layouts (quantized weights read directly from disk via mmap, no copy to RAM needed) and platform-specific SIMD instructions. It turned a 16 GB GPU requirement into a 4 GB RAM requirement for an equivalent 4-bit quantized model. This single project democratised local LLMs almost overnight.

llama.cpp also introduced the GGUF file format (see below) and the quantization naming scheme (Q4_K_M, Q5_K_M etc.) that the entire community now uses. Most quantized models on Hugging Face are prepared specifically for llama.cpp.

From a software-eng POV

llama.cpp is the inference equivalent of writing a database engine from scratch in C instead of using an ORM. You lose the convenience of Python abstractions but gain precise control over memory layout, alignment, and cache behaviour, which matters enormously when your bottleneck is loading 4 GB of weights from DRAM for every forward pass.

GGUF: the universal container format

Before GGUF, models were distributed as directories of files: the weight shards lived in .bin or .safetensors files, and a separate config.json told the engine the model's architecture (number of layers, head dimensions, etc.). This split caused endless "config not found" and version mismatch errors.

GGUF (GPT-Generated Unified Format, introduced 2023) bundles everything into one binary file: the quantized weights, the tokenizer vocabulary, and all architectural metadata. It has two killer features:

  • mmap (memory mapping): the engine maps the file into virtual address space rather than copying it to RAM. The OS pages in only the parts that are actually needed. This makes startup near-instant and reduces peak memory usage significantly.
  • Forward-compatible metadata: new fields can be added to future GGUF versions without breaking older engines that simply ignore unknown keys.
Analogy · a Docker image vs. a bare binary

GGUF is to model weights what a Docker image is to an application. A Docker image bundles the executable, its dependencies, and its configuration into one artifact that runs the same way anywhere. GGUF does the same for a model: one file, all the information the engine needs, portable across every tool that speaks the format.

Ollama: the user-friendly dashboard

If llama.cpp is the engine, Ollama is the dashboard and steering wheel. Using llama.cpp directly requires manually downloading GGUF files from Hugging Face, tracking which quantization level you're using, and passing dozens of command-line flags. Ollama wraps that engine in a clean CLI and a local REST API.

bash · Ollama in three commands 
# 1. Install (macOS)
brew install ollama

# 2. Pull a model — Ollama fetches the right Q4_K_M GGUF automatically
ollama pull llama3.2

# 3. Chat — or hit the OpenAI-compatible REST API
ollama run llama3.2
# or: curl http://localhost:11434/api/chat -d '{"model":"llama3.2","messages":[...]}'

From a software-eng POV

Ollama is a service manager and model registry combined. It handles the full lifecycle: pulling a versioned model image from a registry (like Docker Hub, but for GGUF models), managing memory (loading the model on first request, unloading after idle timeout), and exposing a clean REST API. Other apps (Open WebUI, Continue.dev, LangChain) all connect to Ollama's endpoint rather than managing llama.cpp directly.

vLLM: production-grade GPU serving

While llama.cpp is optimised for efficiency on a single user's machine, production systems need to serve thousands of concurrent requests. This is where vLLM comes in.

vLLM's core innovation is PagedAttention: a technique borrowed directly from how operating systems manage virtual memory. Standard inference pre-allocates a fixed KV cache block per request. This wastes GPU memory for short sequences and fails outright for long ones. PagedAttention divides the KV cache into small, fixed-size pages and allocates them on demand, like OS page tables for process memory. This allows vLLM to pack far more concurrent requests into the same GPU memory budget.

Analogy · OS virtual memory management

Traditional inference is like allocating a fixed-size parking garage bay for every car that might visit, even if most cars are compact, you reserve a full-size space. PagedAttention is like OS virtual memory: you hand out virtual addresses on demand, backed by physical pages only when actually needed. A process thinks it has a contiguous 8 GB address space; a request thinks it has a large KV cache. Neither knows the memory is fragmented and shared at the physical level.

Triton: the GPU kernel compiler

Writing raw CUDA (C++) for GPUs is notoriously difficult. Every new attention variant or quantization scheme requires implementing custom CUDA kernels: low-level GPU programs that can take weeks to write and debug correctly.

Triton (by OpenAI, not the inference server) is a Python-like language that compiles to GPU machine code as fast as hand-written CUDA. It lets researchers write a new attention kernel in Python, and the Triton compiler handles the low-level register allocation, memory coalescing, and thread scheduling.

FlashAttention (the memory-efficient attention algorithm now used by virtually every production model) is implemented in Triton. Without Triton, implementing FlashAttention's complex tiling algorithm in raw CUDA would have been a months-long project accessible only to CUDA experts.

From a software-eng POV

Triton is to GPU programming what high-level languages were to assembly in the 1970s. You describe what you want to compute; the compiler figures out how to lay it out on the GPU's thousands of cores. The result is within a few percent of hand-optimised CUDA speed, at a fraction of the development cost. This is why vLLM and similar engines can ship support for new architectures within days of a model release: they write Triton kernels, not CUDA.

Choosing the right runtime

The decision is almost always determined by your hardware:

  • Mac / no NVIDIA GPU: llama.cpp directly, or Ollama for a zero-config experience. Use Q4_K_M GGUF models.
  • Single NVIDIA GPU (24 GB or less), developer use: Ollama (which can use the GPU automatically) or llama.cpp with CUDA offloading. Alternatively, transformers + bitsandbytes for 4-bit.
  • Multi-GPU production cluster: vLLM. It supports tensor parallelism across GPUs, continuous batching, and the OpenAI API schema out of the box.
  • Embedded / edge / constrained hardware: llama.cpp with aggressive quantization (Q2_K or Q3_K_M), or specialised frameworks like MLC-LLM which compile models to device-specific binaries.

Recap

  • A model file (weights) is inert data. A runtime engine is the program that loads those weights and executes the maths. They are entirely separate.
  • llama.cpp brought LLMs to ordinary hardware by reimplementing the inference maths in C++ with no GPU required. It also created the GGUF format and the k-quant naming scheme.
  • GGUF bundles weights + tokenizer + config into one file, with mmap support for near-instant startup and low peak memory.
  • Ollama wraps llama.cpp in a service manager and OpenAI-compatible REST API: zero configuration, pull-and-run workflow.
  • vLLM uses PagedAttention (inspired by OS virtual memory) to serve thousands of concurrent users on GPU clusters with far higher throughput than naive implementations.
  • Triton is a Python-like GPU kernel language that lets engineers write high-performance GPU code without deep CUDA expertise; it's how FlashAttention was made practical.
