The ML Stack, Demystified

PyTorch, CUDA, Hugging Face, pydantic — these terms form the alien map of the ML world. Each one maps exactly to a role you already know. This plate is your Rosetta Stone.

Reading time · 16 min Prereqs · Plates 01, 02 Code · transformers · pydantic

Why Python won

If you're a backend engineer who grew up on Java, Go, or Rust, the ML ecosystem's commitment to Python might look like an accident of history. It isn't — or rather, it is an accident, but one that created an unassailable moat.

The real computation in deep learning doesn't happen in Python. It happens in highly optimized C++/CUDA kernels inside libraries like PyTorch and cuDNN. Python is the glue layer — the scripting interface that assembles these kernels, loops over batches, and handles data loading. Because Python is just scripting the heavy work, its interpreter overhead is nearly irrelevant: a forward pass on a 7B model is 99.9% GPU kernel time and ~0.1% Python overhead.

This architecture — a thin Python API over a fast native core — turns out to be ideal for research: you can change model architectures interactively in a Jupyter notebook, see results in seconds, and deploy the same code to production without a rewrite. NumPy established the convention; PyTorch followed it. The ML community built its tooling on top, and the gravity well became self-reinforcing. Python is the interface; the metal does the work.

PyTorch: NumPy with a GPU and autograd

PyTorch is the dominant deep learning framework today. If you know NumPy — arrays, slicing, broadcasting — you already know 80% of the API. The key difference is that PyTorch operates on tensors: multi-dimensional arrays that can live on a GPU and carry a computation graph.

Analogy · NumPy + two superpowers

Imagine NumPy arrays with two additions: (1) they can be moved to a GPU with .to("cuda") and all operations run there automatically — 100× faster for large parallel math; (2) every operation is automatically recorded in a computation graph. Call .backward() on any scalar and PyTorch walks that graph in reverse, computing the gradient of every contributing operation via the chain rule (backpropagation). That's the whole secret of training: compute loss, call backward, nudge weights by their gradient. PyTorch automates step two entirely.

TensorFlow (Google) predates PyTorch and was dominant until roughly 2020. It used a "define then run" static graph, which was faster in production but painful for debugging — you couldn't just print() intermediate values. PyTorch's "define by run" eager execution won the research community, and when torch.compile arrived in PyTorch 2.0, it closed most of the production performance gap. TensorFlow still runs production workloads at Google; JAX (also from Google) is the frontier-research alternative that compiles functional transformations to XLA. For this guide, PyTorch is the lingua franca.

The hardware: GPUs, VRAM, CUDA, and Apple's MPS

A CUDA-capable NVIDIA GPU is the industry-standard compute unit for both training and inference. The key resource is VRAM (Video RAM) — the GPU's on-chip memory. Everything the model operates on — the weight matrices, the activations, the key-value cache — must fit in VRAM during a forward pass. If it doesn't fit, the job crashes with an OutOfMemoryError.

The rule of thumb: a model requires roughly 2 bytes per parameter in 16-bit (fp16) precision. A 7B model needs ~14 GB of VRAM just for weights; during training you add activations, optimizer state, and gradients — often 4–6× more. For inference only, 14 GB fits on a single consumer GPU (e.g. RTX 3090/4090). Quantization (Plate 13) can shrink this further: in 4-bit, a 7B model fits in ~4 GB.

Gotcha · CUDA / driver / version mismatches

The most common ML setup failure is a version mismatch between the CUDA toolkit, the NVIDIA driver, and the CUDA build that PyTorch was compiled against. The symptom is cryptic: models import fine but crash the moment they hit the GPU. The fix: always install PyTorch from pytorch.org (not just pip install torch), selecting the exact CUDA version your driver supports. Run nvidia-smi to see your driver's maximum supported CUDA version; install PyTorch built for that or lower. Mismatches at this layer waste hours.

Apple Silicon (M1 through M4 series) uses a unified memory architecture: the CPU and GPU share the same RAM pool. PyTorch accesses the GPU via MPS (Metal Performance Shaders — Apple's GPU compute API, analogous to CUDA). On an M-series MacBook with 32 GB of unified memory you can run 13B-parameter models at reasonable inference speed — a remarkable dev environment that needs no external GPU. Use device = "mps" everywhere you'd otherwise use "cuda".

Cloud GPU instances (A100, H100 on AWS/GCP/Lambda/RunPod) are the right choice for training or running models too large for a local machine. An A100 (80 GB VRAM) can run a 70B model in fp16 on a single card. For quick experiments, T4 (16 GB) and A10G (24 GB) instances are much cheaper and sufficient for most inference.

The Hugging Face universe

Hugging Face is the hub around which almost all open-weights ML practice revolves. Think of it as three things at once:

From a software-eng POV

The Hub (huggingface.co/models) is GitHub + Docker Hub for model weights. A model card is the README; the repository holds the weight files, tokenizer config, and config.json. You pull a model with from_pretrained("org/model-name") — it downloads to ~/.cache/huggingface/hub and is cached locally thereafter. Version tags, branches, and commit hashes work just as in git. Treat a model identifier like an immutable Docker image tag: meta-llama/Llama-3.1-8B-Instruct is a specific artifact, not a floating pointer.

The ecosystem around the Hub is a constellation of libraries. Here are the ones you will encounter:

LibraryRoleSoftware-eng analogy
transformers Load and run pretrained models; tokenizers; pipelines for common tasks The standard SDK — the boto3 of LLMs
datasets Load, stream, and preprocess text corpora; handles sharding and caching A typed, lazy ORM over training data
peft Parameter-efficient fine-tuning: LoRA, QLoRA, adapter layers The fine-tuning middleware (Plate 10)
accelerate Abstract over single-GPU / multi-GPU / multi-node training without changing model code A distributed-systems abstraction layer — like writing code once for both single-thread and thread-pool
tokenizers Fast Rust-backed tokenizer implementation; used internally by transformers A compiled parser — the C extension under the hood of the Python tokenizer API
trl Supervised fine-tuning and RLHF pipelines (SFT, PPO, DPO) The training orchestration layer for alignment (Plate 11)

You don't need to know all of them on day one. Start with transformers for loading and running models, add datasets when you need training data, and reach for peft when you're ready to fine-tune. The rest are there when you need them.

There is also the GGUF format and llama.cpp — a highly optimized C++ inference engine that runs quantized models on CPU (and MPS/Metal) without PyTorch at all. Wrapped by Ollama, this is how many developers run models locally without thinking about Python environments. We'll return to it in Plate 13.

pydantic and the app-layer glue

pydantic shows up constantly in LLM application code and trips up engineers who expect it to be an ML library. It isn't. Pydantic is a Python data validation library — you define a schema using Python type annotations, and pydantic validates, coerces, and serializes JSON payloads against it. Think of it as the Protobuf/JSON Schema of the Python world, but with runtime validation built in.

Its relevance to LLMs is precisely that: when you ask a model to produce structured output (a JSON object with specific fields), you need something to validate the response and convert it into a typed Python object. Pydantic is the standard tool for this. Libraries like Instructor and Outlines use pydantic schemas to constrain or post-process model output.

python · pydantic schema for LLM output 
# pip install pydantic
from pydantic import BaseModel, Field
from typing import Literal
import json

# Define the shape you want the model to emit
class SentimentResult(BaseModel):
    label: Literal["positive", "negative", "neutral"]
    confidence: float = Field(ge=0.0, le=1.0)
    reasoning: str

# Suppose the LLM returned this JSON string:
raw = '{"label":"positive","confidence":0.92,"reasoning":"The review praises the product."}'

# Pydantic validates and types it in one line — raises ValidationError on bad data
result = SentimentResult.model_validate_json(raw)
print(result.label)       # "positive"  (a typed Python string, not raw JSON)
print(result.confidence)  # 0.92        (a float, validated to be in [0, 1])

# Going the other way — generate the JSON schema to put in a system prompt:
print(json.dumps(SentimentResult.model_json_schema(), indent=2))

Pydantic is not doing anything ML-specific. It is runtime type-checking for Python — the kind of thing you'd normally get from a compiled language's type system. In an LLM application, it's the contract layer between the "write some JSON" instruction you send the model and the typed Python object your downstream code expects.

Setting up without tears

The ML Python stack has one genuine pain point: environment management. PyTorch must be compiled for a specific CUDA version. CUDA must match your driver. The wrong pip install can silently install a CPU-only torch and you won't know until your GPU is idle and training takes three hours instead of three minutes.

The reliable setup process, step by step:

  1. Create an isolated environment: python -m venv .venv && source .venv/bin/activate (or conda create -n llms python=3.11). Never install ML libraries into your system Python.
  2. Check your CUDA version: nvidia-smi shows the driver's max CUDA support in the top-right. On Mac, skip this — MPS needs no CUDA.
  3. Install PyTorch from the official selector at pytorch.org/get-started/locally — it gives you the exact pip install command for your OS, CUDA version, and package manager. Do not deviate.
  4. Install Hugging Face libraries: pip install transformers datasets accelerate peft trl.
  5. Verify GPU is visible: 
    python -c "import torch; print(torch.cuda.is_available(), torch.backends.mps.is_available())"
    You should see True False on NVIDIA or False True on Apple Silicon.
python · hello world — load and generate 
# pip install transformers torch
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_id = "Qwen/Qwen2.5-0.5B-Instruct"   # ~1 GB download; runs fine on MPS
device   = "mps"                            # Apple Silicon GPU

tokenizer = AutoTokenizer.from_pretrained(model_id)
model     = AutoModelForCausalLM.from_pretrained(
    model_id, torch_dtype=torch.float16
).to(device)

prompt = "In one sentence, what is a language model?"
inputs = tokenizer(prompt, return_tensors="pt").to(device)

with torch.no_grad():
    out = model.generate(**inputs, max_new_tokens=50, do_sample=False)

print(tokenizer.decode(out[0], skip_special_tokens=True))
# pip install transformers torch accelerate bitsandbytes
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_id = "meta-llama/Llama-3.2-3B-Instruct"   # requires HF token for gated models
device   = "cuda"

tokenizer = AutoTokenizer.from_pretrained(model_id)
model     = AutoModelForCausalLM.from_pretrained(
    model_id,
    torch_dtype=torch.bfloat16,   # bfloat16 is preferred on A100/H100
    device_map="auto",             # distribute across all visible GPUs automatically
)

prompt = "In one sentence, what is a language model?"
inputs = tokenizer(prompt, return_tensors="pt").to(device)

with torch.no_grad():
    out = model.generate(**inputs, max_new_tokens=50, do_sample=False)

print(tokenizer.decode(out[0], skip_special_tokens=True))
output
In one sentence, what is a language model?
A language model is a statistical system trained on large amounts of
text to predict the probability of the next token in a sequence.
# ~1.5 s on M2 Pro (MPS); ~0.3 s on A100 (CUDA)

Notice the structure: download tokenizer and model weights separately (both cached after first run), move the model to the target device, tokenize the input on the same device, run model.generate() inside a torch.no_grad() context (disables gradient tracking — you don't need it for inference, and it saves memory), then decode back to text. This pattern repeats throughout the guide.

▸ What device_map="auto" actually does deep dive

device_map="auto" invokes Hugging Face's accelerate library to inspect the model's layer graph and the available devices (multiple GPUs, CPU RAM, even disk), then distribute the layers to fit. It generates a mapping like {"model.embed_tokens": 0, "model.layers.0": 0, …, "model.layers.20": 1, …}. Data is automatically transferred between devices as the forward pass proceeds. This is how you run a 70B model across two 40 GB A100s — or even CPU-offload layers that don't fit. For inference, device_map="auto" is almost always what you want on multi-GPU setups.

Recap

  • Python is a thin, scriptable interface over fast C++/CUDA kernels — not a performance bottleneck; the GPU is doing the work.
  • PyTorch = NumPy + GPU execution + automatic differentiation. Tensors live on a device; .backward() computes gradients automatically for training.
  • CUDA is NVIDIA's GPU compute layer; MPS is Apple's equivalent. VRAM is the critical resource — ~2 bytes per parameter in fp16. Version mismatches between driver and PyTorch CUDA build are the most common setup failure.
  • The Hugging Face Hub is GitHub + Docker Hub for model weights. Pull a model with from_pretrained(); the weights are cached locally. transformers, datasets, peft, accelerate, and trl are the key libraries that wrap it.
  • pydantic is not an ML library — it is runtime type validation for Python. In LLM apps it is the contract layer between "generate some JSON" and "give me a typed Python object."
  • Environment setup: isolated venv or conda, PyTorch installed from pytorch.org for your exact CUDA version, then Hugging Face libraries. Don't skip the verification step.
