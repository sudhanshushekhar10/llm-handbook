Evaluating Models

Building a model is mostly engineering. Knowing whether it actually works (for your task, in your environment) is the genuinely hard part. Benchmarks lie, perplexity misleads, and if you aren't careful you'll optimize a metric until it stops meaning anything.

Reading time · 18 min Prereqs · Plate 06 Code · lm-evaluation-harness · openai

Why evaluation is the hard part

In most software you know when something is broken. The function returns the wrong value; the test fails red. With language models, the output is text, and text admits infinite variation. "Paris" and "Paris, France" are both correct answers to "What is the capital of France?" but one contains more words. "Write a helpful summary" produces a space of valid outputs so large that no enumeration could cover it. You can't write a unit test for fluency.

The field's answer has been to proxy: measure something correlated with quality that is computable. Perplexity on held-out text. Multiple-choice accuracy on a knowledge benchmark. Pass rate on a coding test suite. Each proxy captures something real, and each has failure modes that let bad models game it: sometimes accidentally, sometimes by design. Understanding the proxy's limits is the whole skill.

From a software-eng POV · evals are your test suite

Think of an eval suite as CI for a model. Before you ship a fine-tune, a RAG update, or a prompt change, you want a regression harness that tells you "this didn't get worse on the things we care about." If you have no eval, you have no confidence; you're merging to main without tests. The benchmark ecosystem provides off-the-shelf suites; your job is to add the task-specific ones only you can write.

Academic benchmarks: MMLU, GSM8K, HumanEval and friends

A handful of benchmarks became the de-facto lingua franca for comparing models. Understanding what each actually measures (and what it doesn't) saves you from being misled by a leaderboard.

Benchmark What it tests Format Watch out for
MMLU World knowledge across 57 subjects (law, medicine, history…) 4-choice multiple choice Saturating near 90%; contamination risk
HellaSwag Commonsense reasoning about everyday events Multiple choice, adversarial Near-solved by big models; ceiling effect
GSM8K Grade-school math word problems requiring multi-step reasoning Free-form, exact numeric answer Contamination; strong zero-shot variance
HumanEval Python function completion, measured by unit-test pass rate Code generation, pass@k Only 164 problems; docstrings may overlap with training
MT-Bench Multi-turn chat quality across 8 categories LLM-graded, 1–10 scale Judge model bias; correlates with Chatbot Arena

MMLU arrived in 2021 and immediately became the default "how smart is the model" number. It's 57 subjects, each with multiple-choice questions drawn from practice exams and textbooks. It's genuinely broad, and early models that scored 70%+ felt impressively knowledgeable. The problem: as models scaled up and as MMLU scores climbed past 85%, it started telling us less and less. Headroom collapsed.

HumanEval introduced pass@k: generate k candidate completions, run the unit tests on all of them, and declare success if at least one passes. This corrects for stochasticity: a model that occasionally gets the right answer but usually doesn't should score lower than one that's reliably correct. It's a clean, objective signal for code tasks.

▸ How pass@k is computed deep dive

If you generate n samples total and c of them pass, the unbiased estimator of pass@k is:

pass@k = 1 − C(n−c, k) / C(n, k)

where C(a,b) is the binomial coefficient. This is more statistically efficient than sampling exactly k times per problem; you run n ≥ k samples once and compute the estimator analytically. Chen et al. used n=200 and report pass@1, pass@10, pass@100.

Perplexity and its limits

Before there were benchmarks, there was perplexity. It's defined as the exponentiated average negative log-likelihood the model assigns to held-out text: essentially, "how surprised was the model by the test corpus?" Lower is better: a model that assigns high probability to what actually happened is less perplexed.

Perplexity is elegant. It doesn't require labeled examples or a judge. You just run the model over text it hasn't seen and measure the loss. It's fast, it's reproducible, and it tracks improvements in language modelling faithfully. For pretraining runs, it's the primary signal.

Its limits emerge the moment you ask "but is the model useful?" A model can have excellent perplexity on Wikipedia and still:

  • Refuse every question it dislikes (aligned to unhelpfulness).
  • Hallucinate confidently; low hallucination is not what perplexity measures.
  • Follow instructions poorly: it's great at completing sentences, terrible at completing tasks.
  • Score worse on code perplexity while being better at coding (distribution mismatch).
Analogy · compression ratio vs. correctness

Perplexity measures how well the model compresses text, like a zip ratio. A model that compresses English prose efficiently does so because it learned the statistics of English prose. But a good compression ratio on the wrong corpus (say, dense legal text) tells you nothing about whether the model will be useful to your end users. Perplexity is a proxy for fluency, not for usefulness.

LLM-as-a-judge and Chatbot Arena

If the problem is that text quality can't be reduced to a scalar, the obvious move is to have something that can understand text evaluate it. Enter LLM-as-a-judge: use a strong model (GPT-4, Claude, Gemini) to grade another model's outputs on a 1–10 scale or in a head-to-head preference comparison.

MT-Bench asked GPT-4 to score multi-turn conversations across eight categories (coding, math, writing, roleplay…). The resulting scores correlated well with human preferences, suggesting that a strong judge model captures much of what humans care about without the cost and slowness of human annotation. This is now standard practice for evaluating chat and instruction-following quality.

Chatbot Arena (LMSYS) takes a different angle: real users submit a question, get responses from two anonymous models, pick the winner. The winner's Elo rating goes up. Over millions of votes, this produces a ranking that is grounded in real user preference rather than any single judge's biases. It's the closest the field has to a ground truth for "which model is better to use."

Gotcha · judge bias and self-preference

LLM judges have systematic biases: they prefer longer outputs, more structured responses, and (in controlled studies) their own outputs. "GPT-4 rates GPT-4 highly" is a real phenomenon. To mitigate this: swap the order of responses (position bias is real), use multiple judge models, and calibrate against human labels on a random subset.

Contamination: the benchmark you trained on

Here's the quiet problem eating the whole field: once a benchmark is published and its questions are on the internet, future training runs will ingest them. A model that has seen "the answer to question 47 of MMLU Jurisprudence is C" during pretraining didn't learn law. It memorized an exam.

This is called benchmark contamination or data leakage, and the community has coined the term benchmaxxing for the practice (intentional or not) of training on or near benchmark data to inflate scores. Scores plateau, papers report state-of-the-art, and nothing improves in practice. It's teaching to the test at industrial scale.

Gotcha · benchmaxxing

A model that scores 88% on MMLU might have seen MMLU questions during pretraining. Models trained on carefully curated, non-overlapping data often score lower on public benchmarks but better on held-out private evals. When a model's benchmark number seems suspiciously high, look for contamination analysis in the technical report. If there isn't one, assume the number is optimistic. Trust private, held-out evals over public leaderboard positions for anything that matters.

The practical response is to use benchmarks as rough filters (enough to tell a 60% model from an 85% model) but to invest seriously in private, held-out evals that were never published. If the questions never hit the internet, they can't leak into the training corpus. This is why leading labs maintain eval sets that are never released publicly.

Alignment evaluations suffer the same problem: if the alignment dataset leaks, the model learns to say the aligned-sounding answer rather than actually being aligned. Chain-of-thought prompting benchmarks are particularly susceptible: a few-shot example of the reasoning style will often appear verbatim in training corpora.

Building evals you can trust

The practical upshot of all the above: you need to write your own evals. No public benchmark tests whether your customer-support model handles the specific product SKUs your users ask about. No leaderboard tells you whether the SFT-trained variant is better at your JSON extraction task than the base model.

Good custom eval design borrows from test engineering:

  • Golden dataset: a hand-curated set of (input, expected-output) pairs that cover your real traffic. 50–200 examples beats 10,000 synthetic ones.
  • Exact-match and F1 where the answer space is small (classification, extraction, named entities).
  • Pass@k or execution-based for code: run the generated code; don't read it.
  • LLM-as-judge for open-ended tasks: write a rubric, test it on cases where you know the answer, then automate it.
  • Regression suite: run the full eval before and after any model change. Flag any metric that dropped by more than 2 percentage points.
Analogy · Goodhart's Law

"When a measure becomes a target, it ceases to be a good measure." This is Goodhart's Law, and it applies to LLM evaluation with brutal clarity. Once a benchmark is the thing being optimized, the benchmark stops measuring what it was supposed to measure. The only defense is to keep rotating your metrics and to maintain held-out, never-public test sets. Think of it as the eval equivalent of not letting developers write their own tests, and then grade their own work.

The EleutherAI lm-evaluation-harness is the open-source standard for running academic benchmarks in a reproducible way. It implements MMLU, HellaSwag, GSM8K, HumanEval, and dozens of others, and it's the harness used by most open-model leaderboards (Open LLM Leaderboard on Hugging Face). Worth knowing even if you mostly write custom evals; it's the reference implementation.

python · custom eval with exact-match & pass@k 
# pip install transformers torch
# Exact-match eval + pass@k for a tiny code-gen model.
import torch, math
from transformers import AutoModelForCausalLM, AutoTokenizer
from itertools import combinations

device = "mps" if torch.backends.mps.is_available() else "cpu"
model_id = "HuggingFaceTB/SmolLM2-360M-Instruct"
tok   = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id, torch_dtype=torch.float16).to(device)

# ── Exact-match QA ──────────────────────────────────────────────────────────
qa_dataset = [
    {"q": "What is 7 * 8?",           "a": "56"},
    {"q": "Capital of France?",        "a": "Paris"},
    {"q": "sqrt(144) =",               "a": "12"},
]

def generate(prompt, max_new=32):
    ids = tok(prompt, return_tensors="pt").input_ids.to(device)
    with torch.inference_mode():
        out = model.generate(ids, max_new_tokens=max_new, do_sample=False)
    return tok.decode(out[0][ids.shape[-1]:], skip_special_tokens=True).strip()

correct = sum(
    generate(f"Answer with just the answer:\n{ex['q']}\nAnswer:").startswith(ex["a"])
    for ex in qa_dataset
)
print(f"Exact-match: {correct}/{len(qa_dataset)} = {correct/len(qa_dataset):.0%}")

# ── pass@k (unbiased estimator) ──────────────────────────────────────────────
def pass_at_k(n: int, c: int, k: int) -> float:
    """Unbiased estimator from Chen et al. 2021."""
    if n - c < k:
        return 1.0
    return 1.0 - math.comb(n - c, k) / math.comb(n, k)

# Simulate: n=5 samples, c=3 passed unit tests
n, c, k = 5, 3, 1
print(f"pass@{k} (n={n}, c={c}): {pass_at_k(n, c, k):.2%}")

For open-ended tasks where exact-match doesn't apply, wire up a judge model with a structured rubric:

python · LLM-as-judge prompt 
# pip install openai
# LLM-as-judge: ask a strong model to score a response on a rubric.
import openai, json

client = openai.OpenAI()  # uses OPENAI_API_KEY env var

JUDGE_SYSTEM = """You are an expert evaluator. Given a user question and a model response,
rate the response on three axes, each 1-5:
  - Accuracy: Is the information factually correct?
  - Helpfulness: Does it directly address what the user needed?
  - Conciseness: Is it appropriately brief without losing important detail?

Return ONLY a JSON object: {"accuracy": N, "helpfulness": N, "conciseness": N, "rationale": "..."}"""

def judge(question: str, response: str) -> dict:
    msg = f"Question: {question}\n\nModel response: {response}"
    result = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": JUDGE_SYSTEM},
            {"role": "user",   "content": msg},
        ],
        temperature=0,
        response_format={"type": "json_object"},
    )
    return json.loads(result.choices[0].message.content)

scores = judge(
    question="Explain what a LoRA adapter is in one sentence.",
    response="LoRA adds small trainable matrices to a frozen model so you fine-tune 0.1% of parameters.",
)
print(scores)
output
{"accuracy": 5, "helpfulness": 5, "conciseness": 5,
 "rationale": "Precisely correct, directly addresses the question, appropriately terse."}

Recap

  • Evaluation is the hard part: text outputs don't have a natural ground truth, so the field uses proxies, each with failure modes.
  • MMLU tests broad knowledge (57 subjects, multiple choice); GSM8K tests multi-step math; HumanEval tests code via unit-test pass rate (pass@k); MT-Bench & Chatbot Arena measure chat quality via preference.
  • Perplexity is an excellent signal for pretraining progress but a poor judge of downstream usefulness: a fluent model can still be wrong, unhelpful, or hallucinating.
  • LLM-as-a-judge (use a strong model to score outputs against a rubric) correlates well with human preference and scales cheaply; watch for position bias and self-preference.
  • Contamination (benchmark questions appearing in training data) inflates public scores. Treat leaderboard numbers as rough filters; run held-out private evals for anything that matters.
  • Build task-specific evals: a golden dataset, exact-match or execution-based scoring where possible, LLM-judge for open-ended, and a regression harness you run on every model change. Goodhart's Law is always watching.

With solid evals in hand, you're ready to tackle retrieval-augmented generation, giving the model access to fresh knowledge at inference time. On to Plate 17.
