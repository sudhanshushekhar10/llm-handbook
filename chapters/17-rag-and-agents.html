RAG, Tools & Agents

A model trained last year doesn't know about your internal docs, your database, or what happened this morning. RAG and tool use are how you give it access — without retraining. Understanding how they actually work turns you from a prompt engineer into an LLM systems engineer.

Reading time · 20 min Prereqs · Plates 04, 06, 09 Code · transformers · sentence-transformers · FAISS · pydantic

The knowledge gap problem

Every LLM has a knowledge cutoff and a context limit. It doesn't know about your company's Q4 earnings call, your internal API docs, or the bug report filed this morning. It also can't run code, look up live prices, or write to your database. Two complementary solutions emerged:

  • RAG (Retrieval-Augmented Generation): retrieve relevant text and paste it into the context before asking the model to answer. Give it the reference manual instead of teaching it the content.
  • Tool use / function calling: let the model emit a structured request to call an external function (search API, calculator, database query) and receive the result — turning the model into an orchestrator.
Analogy · open-book exam vs. delegation

RAG is an open-book exam: the student (model) can't memorize everything, so you hand them the right pages before each question. Tool use is more like delegation: the student writes a memo ("please look up the current stock price and report back") and incorporates the response. Both are runtime knowledge augmentation, not training.

The RAG pipeline

RAG has two phases that mirror a standard database architecture:

Indexing (offline, once): Split your documents into chunks. Embed each chunk into a dense vector using an embedding model. Store the vectors in a vector database (FAISS, Pinecone, Weaviate, pgvector…).

Retrieval + generation (online, per query): Embed the user's query using the same embedding model. Find the top-k most similar document chunks by cosine similarity (or ANN search). Stuff those chunks into the model's prompt. Ask the model to answer using the provided context.

Embedding models: the semantic compass

An embedding model maps a piece of text to a dense vector — a point in a high-dimensional space where similar meaning ≈ nearby points. The embedding model is separate from (and usually much smaller than) your generative LLM.

Popular choices: text-embedding-3-small (OpenAI API, 1536 dims), all-MiniLM-L6-v2 (fast, 384 dims, runs locally), BAAI/bge-m3 (multilingual, 1024 dims, very strong). For production, run the same model family for both indexing and querying — mixing models gives gibberish similarity scores.

python · full RAG pipeline from scratch 
# pip install sentence-transformers faiss-cpu ollama
import faiss, numpy as np
from sentence_transformers import SentenceTransformer

# 1. Index phase: embed some documents
docs = [
    "The Python GIL prevents true multi-threading for CPU-bound tasks.",
    "asyncio is for I/O-bound concurrency, not CPU parallelism.",
    "Use multiprocessing for CPU-bound parallel Python work.",
    "FAISS is a library for efficient similarity search over dense vectors.",
]
embedder = SentenceTransformer("all-MiniLM-L6-v2")
doc_vecs = embedder.encode(docs, normalize_embeddings=True)  # shape: [4, 384]

index = faiss.IndexFlatIP(doc_vecs.shape[1])   # cosine via inner product
index.add(doc_vecs.astype("float32"))

# 2. Query phase: embed query, retrieve, generate
query = "How do I run things in parallel in Python?"
q_vec = embedder.encode([query], normalize_embeddings=True).astype("float32")
scores, ids = index.search(q_vec, k=2)                  # top-2

context = "\n\n".join(docs[i] for i in ids[0])
prompt = f"Use the context below to answer the question.\n\nContext:\n{context}\n\nQuestion: {query}\nAnswer:"

# Generate with a local model via Ollama
import ollama
resp = ollama.generate(model="llama3.2", prompt=prompt)
print(resp["response"])

import faiss, numpy as np, torch
from sentence_transformers import SentenceTransformer
from transformers import pipeline

docs = [
    "The Python GIL prevents true multi-threading for CPU-bound tasks.",
    "asyncio is for I/O-bound concurrency, not CPU parallelism.",
    "Use multiprocessing for CPU-bound parallel Python work.",
    "FAISS is a library for efficient similarity search over dense vectors.",
]

embedder = SentenceTransformer("BAAI/bge-m3", device="cuda")
doc_vecs = embedder.encode(docs, normalize_embeddings=True)
index = faiss.IndexFlatIP(doc_vecs.shape[1])
res = faiss.StandardGpuResources()
gpu_index = faiss.index_cpu_to_gpu(res, 0, index)
gpu_index.add(doc_vecs.astype("float32"))

query = "How do I run things in parallel in Python?"
q_vec = embedder.encode([query], normalize_embeddings=True).astype("float32")
scores, ids = gpu_index.search(q_vec, k=2)
context = "\n\n".join(docs[i] for i in ids[0])

gen = pipeline("text-generation", model="meta-llama/Llama-3.1-8B-Instruct",
               device_map="auto", torch_dtype=torch.bfloat16)
prompt = f"Context:\n{context}\n\nQuestion: {query}\nAnswer:"
print(gen(prompt, max_new_tokens=150)[0]["generated_text"])

Function calling and tool use

Modern chat models (GPT-4, Claude 3, Llama 3.1) can emit structured function calls instead of free text. You declare the available tools (name, description, parameter schema) in the system prompt; the model decides when and how to call them; your code executes the call and returns the result; the model continues.

The key insight: the model is not actually calling anything. It's generating a JSON blob that you execute. The model is the planner; you are the runtime.

python · tool use with pydantic validation 
# pip install anthropic pydantic
import anthropic, json
from pydantic import BaseModel

client = anthropic.Anthropic()

# Define a tool schema (what the model can call)
tools = [{
    "name": "get_weather",
    "description": "Get current weather for a city.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "City name"},
            "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
        },
        "required": ["city"]
    }
}]

def get_weather(city: str, unit: str = "celsius") -> dict:
    # In reality: call a weather API
    return {"city": city, "temp": 22, "unit": unit, "condition": "sunny"}

# Agentic loop: model may call tools, we respond, model continues
messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]

while True:
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=1024, tools=tools, messages=messages
    )
    if resp.stop_reason == "end_turn":
        print(resp.content[0].text)
        break
    # Handle tool call
    for block in resp.content:
        if block.type == "tool_use":
            result = get_weather(**block.input)
            messages += [
                {"role": "assistant", "content": resp.content},
                {"role": "user", "content": [{
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps(result)
                }]}
            ]
# pip install openai pydantic
from openai import OpenAI
from pydantic import BaseModel
import json

client = OpenAI()

# Pydantic schema → JSON schema for the tool definition
class WeatherInput(BaseModel):
    city: str
    unit: str = "celsius"

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": WeatherInput.model_json_schema()
    }
}]

def get_weather(city: str, unit: str = "celsius") -> dict:
    return {"city": city, "temp": 22, "unit": unit, "condition": "sunny"}

messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]

while True:
    resp = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools
    )
    msg = resp.choices[0].message
    if not msg.tool_calls:
        print(msg.content)
        break
    messages.append(msg)
    for call in msg.tool_calls:
        args = WeatherInput.model_validate_json(call.function.arguments)
        result = get_weather(**args.model_dump())
        messages.append({
            "role": "tool", "tool_call_id": call.id,
            "content": json.dumps(result)
        })
output
# Model emits a tool call (not shown to end user):
# {"name": "get_weather", "input": {"city": "Tokyo", "unit": "celsius"}}

# After tool result injected back:
The current weather in Tokyo is 22°C and sunny.
From a software-eng POV

Tool use is essentially RPC over text. The model generates a structured call; you dispatch it via your normal code; you serialize the response back. The "agentic loop" is just a while loop: keep calling the model until it's done emitting tool calls. The model is a stateless function — all state lives in the messages array you maintain.

Structured outputs with Pydantic

Sometimes you don't want free text at all — you want JSON that matches a schema. Pydantic is the standard tool for this in the Python LLM ecosystem: define your schema as a Pydantic model, generate its JSON Schema, pass it to the model, validate the response.

python · structured extraction with Pydantic 
# pip install anthropic pydantic
from anthropic import Anthropic
from pydantic import BaseModel
from typing import List
import json

class BugReport(BaseModel):
    title: str
    severity: str           # "low" | "medium" | "high" | "critical"
    affected_components: List[str]
    steps_to_reproduce: List[str]
    suggested_fix: str | None = None

client = Anthropic()
raw_text = """
We're seeing random 500s on /api/checkout after the deploy yesterday.
Happens ~5% of the time under load. Stack trace shows NullPointerException
in PaymentService.processCard(). Only affects credit card payments, not PayPal.
"""

resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=f"Extract a structured bug report. Respond with JSON matching this schema:\n{json.dumps(BugReport.model_json_schema(), indent=2)}",
    messages=[{"role": "user", "content": raw_text}]
)
report = BugReport.model_validate_json(resp.content[0].text)
print(f"Severity: {report.severity}")
print(f"Components: {report.affected_components}")
output
Severity: high
Components: ['PaymentService', 'checkout API']
# Full BugReport object is now validated Python — no manual JSON parsing.
Gotcha · constrained decoding vs. prompt-only

Asking a model nicely to "output valid JSON" works most of the time. But for production systems, use constrained decoding (grammar-based token masking) via tools like outlines, guidance, or the response_format={"type": "json_object"} flag on OpenAI — these guarantee syntactically valid output by construction, not by luck.

Agents: loops, memory, planning

An agent is a model in a loop with tools. Rather than a single prompt → response, the agent gets a goal, plans steps, calls tools, observes results, revises its plan, and iterates — all autonomously, until it decides it's done (or hits a step limit).

The minimal agent loop looks like this:

python · minimal ReAct agent pattern 
import anthropic, subprocess, json

client = anthropic.Anthropic()

def run_bash(command: str) -> str:
    result = subprocess.run(command, shell=True, capture_output=True, text=True, timeout=10)
    return result.stdout + result.stderr

tools = [{
    "name": "bash",
    "description": "Run a bash command. Use for file operations, data processing, calculations.",
    "input_schema": {
        "type": "object",
        "properties": {"command": {"type": "string"}},
        "required": ["command"]
    }
}]

SYSTEM = """You are a helpful agent with access to a bash shell.
Think step by step. Use tools to accomplish tasks. When done, respond with FINAL ANSWER: ."""

goal = "Count the number of Python files in /usr/lib and report the total."
messages = [{"role": "user", "content": goal}]

for step in range(10):   # safety limit
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=1024,
        system=SYSTEM, tools=tools, messages=messages
    )
    if resp.stop_reason == "end_turn":
        text = next(b.text for b in resp.content if hasattr(b, "text"))
        if "FINAL ANSWER:" in text:
            print(text.split("FINAL ANSWER:")[-1].strip())
            break
    messages.append({"role": "assistant", "content": resp.content})
    for block in resp.content:
        if block.type == "tool_use":
            output = run_bash(block.input["command"])
            messages.append({"role": "user", "content": [{
                "type": "tool_result", "tool_use_id": block.id, "content": output
            }]})
output
# Agent reasoning trace:
# Step 1 → runs: find /usr/lib -name "*.py" | wc -l
# Step 2 → observes: 4,271
# Step 3 → emits: FINAL ANSWER: There are 4,271 Python files in /usr/lib.
What goes wrong with agents

Agents fail in specific, predictable ways: they get stuck in loops (calling the same tool repeatedly), overshoot their goals (doing more than asked), hallucinate tool results (especially when the tool output is ambiguous), or fail silently when a tool errors. Defensive engineering matters: always add a step limit, log every tool call, validate outputs, and consider giving the agent a "I don't know" or "I'm stuck" tool it can call to exit gracefully.

RAG failure modes to know

RAG sounds simple but has a long tail of failure modes every production system hits:

Failure modeSymptomFix
Chunking mismatchRetrieved chunks miss key context (split mid-sentence)Semantic/paragraph chunking; add chunk overlap
Wrong embedding modelIrrelevant docs retrievedUse domain-appropriate model; BGE-M3 for multilingual
Index stalenessOutdated info despite new docsIncremental re-indexing on document updates
Context window overflowTop-k too large, prompt too longCompress/summarize retrieved chunks; re-rank then select
Hallucination on retrieval missModel answers even when nothing relevant foundCalibrate with "I don't have information about X" fallback

Recap

  • RAG = embed your documents → store in a vector DB → at query time, retrieve the top-k chunks and paste them into the prompt. The model never needs to be retrained.
  • Embedding models map text to dense vectors where semantic similarity = geometric proximity. Always use the same model for indexing and querying.
  • Function calling / tool use lets the model emit structured JSON calls that your code executes. The model is the planner; you are the runtime. The "agentic loop" is just a while loop over model calls.
  • Pydantic + JSON Schema is the standard way to get validated structured output from LLMs. For production, prefer constrained decoding over prompt-only approaches.
  • Agents = model + tools + loop. They fail in predictable ways — always add step limits, log all tool calls, validate outputs.
