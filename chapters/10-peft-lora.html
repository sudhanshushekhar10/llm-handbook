Plate 10 · Part IV — Adapting Models

PEFT & LoRA

Full fine-tuning of a 7B model can demand 100+ GB of GPU memory. LoRA does the same job by freezing the model and training a set of tiny "patch" matrices — under 1% of the parameters. This is the single most important practical technique in the open-model world.

Reading time · 20 min Prereqs · Plates 06, 08, 09 Code · peft · bitsandbytes · trl Widget · VRAM calculator

The problem: full fine-tuning is a memory monster

Fine-tuning means continuing training on your own data. The naive way — update every weight — is called full fine-tuning, and its memory cost is brutal. It's not just the weights. For each parameter you also store a gradient and, for the AdamW optimizer, two more "moment" buffers (plus an fp32 master copy). The rule of thumb is roughly 16 bytes per parameter just for the model state — before activations.

Gotcha · the 16× rule

An 8-billion-parameter model is ~16 GB in fp16 just to hold. To train it fully: 8B × 16 bytes ≈ 128 GB of optimizer/gradient/weight state, plus activations. That's multiple A100s for one small model. Try it in the calculator below — switch Mode to "full fine-tune" and watch the number explode.

The idea: don't move the mountain

Parameter-Efficient Fine-Tuning (PEFT) asks a subversive question: do we really need to update all the weights? Pretraining already taught the model language and world knowledge. Adapting it to a new style or task is a small change. So: freeze the giant pretrained model, and train only a tiny number of new parameters. No gradients or optimizer states for the frozen 8B — only for the small new part.

The first widely used PEFT method was the adapter: insert small trainable bottleneck modules between the frozen layers. It worked, but it added a little latency at inference. The method that took over fixed that.

LoRA: training a patch, not a fork

LoRA — Low-Rank Adaptation — rests on an observation: the change a fine-tune makes to a weight matrix is "low rank." It doesn't need the full expressive power of a dense update; a thin one suffices. So instead of learning a full update matrix ΔW (which is as big as W), LoRA learns two skinny matrices A and B whose product approximates it:

Weffective = Wfrozen + (α / r) · B·A     where A is r×d, B is d×r, r ≪ d

The original W never changes. Only A and B are trained. With a rank r of 8 or 16 against a dimension d of 4096, you're training a fraction of a percent of the parameters — and you only pay optimizer/gradient memory on those.

From a software-eng POV · LoRA is a git patch

The base model is a huge, read-only dependency at a pinned version. A LoRA adapter is a diff against it — a few megabytes that say "add this delta to these matrices." You can keep one base model in memory and hot-swap many adapters (a customer-support patch, a SQL patch, a JSON-mode patch) on top of it, like applying different feature branches to the same checkout. At deploy time you can even merge the patch into the weights so there's zero extra inference cost.

Why LoRA won over adapters

Because B·A can be folded back into W after training (it's just addition), a merged LoRA model has identical architecture and identical inference speed to the original — no extra modules, no latency. You get the storage and memory savings during training and pay nothing at serving time. That combination is why LoRA is everywhere.

▸ rank, alpha, and target modules — the knobs that matter deep dive
  • rank (r): capacity of the adapter. 8–16 is plenty for style/format tasks; 32–64 for teaching substantial new behavior. Higher r = more trainable params and memory.
  • alpha (α): a scaling factor on the update; the effective scale is α/r. A common heuristic is α = 2r. Think of it as a learning-rate-like gain on the adapter.
  • target_modules: which weight matrices get adapters. Classic LoRA targeted only the attention query/value projections; modern practice often targets all linear layers (q,k,v,o and the MLP) for quality.
  • dropout: light regularization (0.05–0.1) on the adapter to reduce overfitting on small datasets.

QLoRA: 4-bit base, full-size model on one GPU

LoRA shrinks the trainable memory, but you still have to hold the frozen base model. For a 70B model that's still 140 GB in fp16. QLoRA closes the gap: quantize the frozen base to 4-bit (Plate 13), so it barely uses memory, and train the LoRA adapters in full precision on top. The base is read-only anyway — quantizing it for the forward pass costs little quality.

The result that changed the hobbyist world

QLoRA made it possible to fine-tune a 65B model on a single 48 GB GPU — and 7B models on free Colab tiers. Combined with 4-bit NF4 quantization and "double quantization," it collapsed the cost of customizing frontier-scale open models from a datacenter to a single card. Set the calculator above to "LoRA / QLoRA" + "int4" to see it.

Hands-on: fine-tune with PEFT

The Hugging Face peft library makes this a few lines. Start tiny and local to learn the moving parts; move to the cloud recipe for a real 8B QLoRA run.

python · LoRA fine-tuning 
# pip install torch transformers peft datasets accelerate
# Runs on an Apple-Silicon Mac (MPS) or CPU with a deliberately tiny model.
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model

device = "mps" if torch.backends.mps.is_available() else "cpu"
base = "HuggingFaceTB/SmolLM2-135M"                 # ~135M params: laptop-friendly
tok  = AutoTokenizer.from_pretrained(base)
model = AutoModelForCausalLM.from_pretrained(base, torch_dtype=torch.float16).to(device)

lora = LoraConfig(
    r=8, lora_alpha=16, lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],           # adapt attention query & value
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora)
model.print_trainable_parameters()
# → trainable params are a tiny fraction of the total. Now run a normal training loop.
# pip install torch transformers peft bitsandbytes trl datasets accelerate
# Needs an NVIDIA GPU (Colab T4/A100). Fine-tunes Llama-3 8B in 4-bit with QLoRA.
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig
from trl import SFTTrainer, SFTConfig
from datasets import load_dataset

base = "meta-llama/Llama-3.1-8B"
bnb = BitsAndBytesConfig(                            # 4-bit NF4 quantization of the frozen base
    load_in_4bit=True, bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True,
)
tok   = AutoTokenizer.from_pretrained(base)
model = AutoModelForCausalLM.from_pretrained(base, quantization_config=bnb, device_map="auto")

lora = LoraConfig(r=16, lora_alpha=32, lora_dropout=0.05,
                  target_modules=["q_proj","k_proj","v_proj","o_proj"], task_type="CAUSAL_LM")

ds = load_dataset("mlabonne/guanaco-llama2-1k", split="train")
trainer = SFTTrainer(
    model=model, train_dataset=ds, peft_config=lora,
    args=SFTConfig(output_dir="out", per_device_train_batch_size=4,
                   gradient_accumulation_steps=4, num_train_epochs=1,
                   learning_rate=2e-4, bf16=True, logging_steps=10),
)
trainer.train()
trainer.save_model("my-lora-adapter")               # saves ONLY the adapter — a few MB
output (local)
trainable params: 147,456 || all params: 134,662,080 || trainable%: 0.1095
# ~0.1% of the model is being trained. That is the entire point of LoRA.

Using your adapter

After training you have a tiny adapter directory. Two ways to use it:

  • Load on top of the base at runtime (PeftModel.from_pretrained(base, "my-lora-adapter")) — keeps the base shared, lets you swap adapters.
  • Merge and export (model.merge_and_unload()) — folds the patch into the weights for a standalone model with zero adapter overhead, ready to quantize and serve (Plates 13–14).
Gotcha · use the right chat template

The #1 cause of disappointing fine-tunes isn't LoRA settings — it's formatting. Your training examples must use the model's exact chat template (the special role tokens it was aligned with). Mismatched templates teach the model the wrong structure. More on this in Plate 12.

Recap

  • Full fine-tuning costs ~16 bytes/param of state — prohibitive for large models.
  • PEFT freezes the model and trains a tiny set of new parameters, eliminating most gradient/optimizer memory.
  • LoRA learns low-rank B·A "patch" matrices; they merge back into the weights, so inference is unchanged. Think git diff over a pinned dependency.
  • QLoRA quantizes the frozen base to 4-bit, letting you fine-tune very large models on a single GPU.
  • Tune rank, alpha, target_modules — and get the chat template right.
