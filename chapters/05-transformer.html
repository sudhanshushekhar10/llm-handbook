Plate 05 · Part II — How LLMs Work

The Transformer

One architecture sits underneath ChatGPT, Llama, Claude, and almost everything else. It replaced a decade of recurrent networks with a single, parallelizable idea: attention. Here is how it actually works, and why it won.

Reading time · 18 min Prereqs · Plates 04, 06 Code · PyTorch Widget · Attention heatmap

The bottleneck recurrence couldn't cross

Before 2017, the state of the art in sequence modelling was the recurrent neural network (RNN), and its better-behaved cousin the LSTM. The mental model is a for loop: read one token, update a hidden state, read the next token, update again. The hidden state is the network's entire memory of everything it has seen so far, a single fixed-size vector carried forward step by step.

This had two fatal problems. First, it is inherently sequential: to compute step 1,000 you must first compute steps 1 through 999. You cannot parallelize across the sequence, which is exactly the thing GPUs are good at. Second, that single carried-forward vector is a bottleneck; by the time a long sentence ends, the beginning has been squeezed and overwritten. The 2014 sequence-to-sequence work made translation work, but long inputs degraded badly.

From a software-eng POV

An RNN is a stream processor with a single mutable accumulator, like folding a list with one register of state. It's correct, but it can't be sharded across workers (each step depends on the last), and the accumulator loses early information. The Transformer is the move to a random-access model: every token can read every other token directly, and the whole thing is one big parallel matrix multiply.

The first crack in the wall was attention, introduced for translation in 2014. Instead of cramming the source sentence into one vector, the decoder was allowed to "look back" at all source positions and decide which were relevant for the word it was currently producing. It was a bolt-on to the RNN. The 2017 insight was radical in its simplicity: if attention is doing the real work, throw the recurrence away entirely. That paper was titled, with justified swagger, Attention Is All You Need.

The one idea: attention as a soft lookup

Here is the whole trick, in terms you already know. Imagine a dictionary (a hashmap). You have a query, and you compare it against every key; wherever the key matches, you return the associated value. A normal hashmap does a hard lookup: exactly one key matches, you get exactly one value.

Attention is a soft, differentiable hashmap. The query is compared against all keys, producing a similarity score for each. Those scores are turned into weights that sum to 1 (via softmax), and the result is a weighted blend of all the values, mostly the ones whose keys matched best. Nothing is discrete, so the whole operation is differentiable and can be learned by gradient descent.

Analogy · the database join

Think of a SQL join where, instead of key = key (exact match), the join condition is a similarity score, and instead of returning matching rows you return a weighted average of all rows. "Query, Key, Value" is not a coincidence of naming: attention really is a fuzzy, learned lookup table.

Self-attention, step by step

In an LLM, the queries, keys, and values all come from the same sequence, hence self-attention. Each token's embedding (from Plate 04) is projected three ways, by three learned weight matrices, into a query vector, a key vector, and a value vector. Then every token's query is dotted against every token's key to get raw scores, those are scaled and softmaxed into weights, and each token's output is the weighted sum of all values.

python · self-attention 
import torch
import torch.nn.functional as F

def self_attention(x, Wq, Wk, Wv):
    # x: (seq_len, d_model) — one row per token
    Q, K, V = x @ Wq, x @ Wk, x @ Wv         # project into queries, keys, values
    d_k = Q.shape[-1]
    scores = (Q @ K.transpose(-2, -1)) / d_k**0.5   # token-to-token similarity, scaled
    # causal mask: token i must not see tokens j > i (no peeking at the future)
    mask = torch.triu(torch.ones_like(scores), diagonal=1).bool()
    scores = scores.masked_fill(mask, float("-inf"))
    weights = F.softmax(scores, dim=-1)      # each row now sums to 1
    return weights @ V                       # blend the values by attention weight

seq, d_model = 5, 16
x  = torch.randn(seq, d_model)
Wq = Wk = Wv = torch.randn(d_model, d_model) * 0.1   # learned during training
out = self_attention(x, Wq, Wk, Wv)
print(out.shape)
import torch
import torch.nn as nn

# The real thing, batched and multi-head, in one line of PyTorch:

x = torch.randn(1, 5, 16)                  # (batch, seq_len, d_model)
attn_out, attn_weights = mha(x, x, x)      # self-attention => query = key = value = x
print(attn_out.shape)                      # the blended representations

That / d_k**0.5 is the "scaled" in scaled dot-product attention. Without it, dot products grow with dimension, pushing softmax into saturated regions where gradients vanish. A tiny detail with a big effect: the kind of thing that separates a paper that works from one that doesn't.

▸ The equation, and why softmax deep dive

The canonical formula is just the code above, compressed:

Attention(Q, K, V) = softmax( Q·Kᵀ / √dk ) · V

Q·Kᵀ is an (n × n) matrix of every-token-against-every-token similarity; this is the quadratic cost of attention, the reason context windows are expensive and the target of a decade of optimizations (FlashAttention, sparse and linear attention, and the state-space models of Plate 15). Softmax converts raw scores into a probability distribution: large scores dominate, but every token keeps a non-zero share, which keeps gradients flowing to everything during training.

Multi-head attention

One attention operation learns one notion of "relevance." But language has many simultaneous relationships: subject–verb agreement, what a pronoun refers to, positional proximity, topical similarity. So we run several attention operations in parallel (multi-head attention), each with its own learned Q/K/V projections operating in a smaller subspace, then concatenate their outputs.

Analogy · a panel of specialists

One head might track "the most recent noun," another "the matching bracket," another "the sentence's subject." It's like running several indexes over the same table, each optimized for a different query pattern, then merging the results. In the widget above, each head is a different lens on the same sentence.

Modern models shave the cost of this with Grouped-Query Attention, which shares keys and values across heads to shrink the KV cache at inference time; we'll meet that in Plates 14 and 15.

The rest of the block

Attention mixes information between tokens. A complete Transformer block wraps it with three more pieces:

  • Positional encoding. Attention is order-blind. Shuffle the tokens and the math is unchanged. So we inject position information into the embeddings. Today this is usually RoPE, which rotates query/key vectors by an angle proportional to position and extends gracefully to long contexts.
  • Residual connections + LayerNorm. Each sub-layer's output is added to its input (a residual/skip connection) and normalized. This is what lets us stack dozens of layers without the signal exploding or vanishing: the same trick that made very deep vision networks trainable.
  • Feed-forward network (FFN). After tokens mix via attention, each token is independently pushed through a small two-layer MLP. This is where much of a model's raw "knowledge" capacity lives, and the part that Mixture-of-Experts later makes sparse.

Stacking up: encoder, decoder, and why GPT-style won

The 2017 architecture had two halves: an encoder (reads the whole input at once, bidirectional) and a decoder (generates output left-to-right, masked so it can't see the future). Three families grew from this:

FamilyShapeSeesBest atExample
Encoder-onlyBidirectionalWhole input at onceUnderstanding / classification / embeddingsBERT
Decoder-onlyCausal (masked)Only the pastGeneration, i.e. being a chatbotGPT, Llama
Encoder–decoderBothInput fully, output causallyTranslation / summarizationT5, the original Transformer
What won, and why

Decoder-only models took over. A single objective (predict the next token) on raw web text scales beautifully, requires no labelled data, and (surprisingly) produces models that can also classify, translate, and summarize when simply asked. One architecture, one objective, all tasks. BERT-style encoders are still the right tool for embeddings and retrieval (Plates 04 & 17), but the chatbots you know are decoder-only.

What didn't, and why

Pure recurrence (RNN/LSTM) faded because it couldn't be parallelized across the sequence and lost long-range information. Ironically, efficiency pressure has revived recurrence-like ideas: state-space models such as Mamba reintroduce linear-time sequential processing, but with modern tricks that fix the old problems. The pendulum swings; the lessons compound.

Recap

  • Attention is a soft, differentiable lookup: compare a query to all keys, return a weighted blend of values.
  • Self-attention lets every token gather context from every other token in parallel: the death of the RNN bottleneck.
  • Multi-head runs many such lookups at once; positional encoding, residuals + LayerNorm, and a per-token FFN complete the block.
  • Stack N blocks → a model. Decoder-only (GPT-style) became the dominant design for generation.
  • The expensive part is attention's quadratic token-to-token matrix, the thing Plates 13–15 spend their time taming.
