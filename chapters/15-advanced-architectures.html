Plate 15 · Part VI — Advanced Architectures

Beyond the Vanilla Transformer

The 2017 Transformer was a brilliant blueprint, but it came with expensive defaults: every token attends to every other token, every parameter activates for every input. Modern architectures surgically fix these bottlenecks, and understanding the fixes tells you a lot about where the field is going.

Reading time · 18 min Prereqs · Plates 05, 13, 14 Code · transformers · llama.cpp

The vanilla Transformer's expensive habits

The 2017 "Attention is All You Need" architecture made two choices that looked fine at 65M parameters but become brutal at 70B:

  • Full attention: every query attends to every key. Cost is O(n²) in sequence length: double the context, quadruple the memory and compute.
  • Dense feed-forward: all parameters fire for every token. You're paying for a 70B-parameter model on every single token, even when most of those parameters are irrelevant to the current input.

Modern architectures attack each problem separately. The result is a family of models that are faster, cheaper, longer-range, and in some cases genuinely smarter, while still being Transformer-based at heart. Let's walk through the big ideas.

Grouped Query Attention: fewer KV heads, same quality

Recall from Plate 05 that multi-head attention runs h independent attention heads. In standard MHA, each head has its own Q, K, V projection. The KV cache stores one K and V tensor per head, per layer, per token in the sequence. At 32 heads and a 128K context, that's enormous.

Multi-Query Attention (MQA) (proposed in the Shazeer 2019 fast decoding paper) uses one shared K and one shared V across all query heads. Memory shrinks by h×, decode is fast, but quality can slip on subtle tasks. Grouped Query Attention (GQA) splits the difference: group the h query heads into g groups, one K/V pair per group.

From a software-eng POV

Think of GQA as connection pooling for attention. Each "query connection" doesn't need its own dedicated K/V socket; multiple queries share the same cached K/V pair. The KV cache is the bottleneck resource at long contexts, and GQA is the pooling layer that lets you serve more queries from the same memory budget.

What won, and why

GQA is now the default in Llama 3, Mistral, Gemma, Phi-3, and most production models released since 2023. It shrinks KV cache memory by 4–8× with minimal quality loss on standard benchmarks. MQA was too aggressive; MHA is too expensive. GQA found the sweet spot.

RoPE: rotary embeddings and long context

How does a model know token position? The vanilla Transformer used learned positional encoding: add a fixed vector to each token embedding before the first layer. It worked, but it was hard to extrapolate to lengths not seen during training.

Rotary Position Embedding (RoPE) rotates the Q and K vectors by an angle proportional to position before the dot product. The rotation is applied in pairs of dimensions; when you take the dot product Q·K, the result depends only on the relative distance between the two positions. No absolute position is ever explicitly encoded.

Analogy · phase angles in signal processing

Imagine two radio signals. Their frequencies encode where they are in time; when you correlate two signals, you get a phase difference that tells you how far apart they were, not where each started. RoPE is the same idea: the "frequency" encodes absolute position, the dot product measures relative offset. This is why RoPE generalises to unseen context lengths more gracefully than additive positional encoding.

To push beyond training context lengths, labs use RoPE scaling: divide all the rotation frequencies by a constant factor (linear scaling) or use more sophisticated non-linear schemes (YaRN, NTK-aware interpolation). Llama 3.1's 128K context window was achieved via a YaRN-like extension on top of RoPE trained at 8K.

Mixture of Experts: sparse parameters

The feed-forward layer is where most Transformer parameters live. In a standard model, every token passes through every FFN parameter: dense and expensive. Mixture of Experts (MoE) replaces the single FFN with N expert networks and a router.

The router is a small linear layer that scores each expert for the current token, then selects the top-k (usually 2–4 out of 8–64). Only those experts compute; the rest sleep. The model has the capacity of a large dense model but the compute cost of a much smaller one.

Key paper
2021
Switch Transformers: Scaling to Trillion Parameter Models with Simple and Efficient Sparsity Fedus, Zoph, Shazeer (Google Brain)

Demonstrated that routing to a single expert per token (top-1) scales cleanly to trillion-parameter models with constant compute. Spawned the modern MoE wave. See reference [21] →

Analogy · call-centre routing

A large call centre has specialists (billing, tech support, complaints). Every call goes to the router (the IVR), which picks the right expert or two for this caller. The company has all the expertise of 50 specialists but each call only activates 2. That's MoE: massive capacity, cheap per-query.

Gotcha · "8×7B" doesn't mean 56B active parameters

Mixtral 8×7B has ~47B total parameters (routed FFNs + shared attention). Each token activates ~2 experts, so the active parameter count during a forward pass is roughly 12–13B, similar to a dense 13B model's compute cost. The "47B" model runs more like a 12B model in practice. Don't confuse total parameters with activated parameters in MoE.

MoE in practice: loading all experts

The catch: even though only a fraction of parameters activate per token, all experts still need to be in memory (you don't know in advance which expert the router will pick). A Mixtral 8×7B model requires ~47B parameters × 2 bytes = ~94 GB in fp16. You need a multi-GPU setup or heavy quantization just to load it.

State Space Models: the challenger

All the architectures above are still attention-based. But there's a fundamentally different approach growing in popularity: State Space Models (SSMs), with Mamba as the flagship.

SSMs replace attention with a recurrence over a compressed hidden state. Each token updates a fixed-size state vector instead of attending over all previous tokens. The key insight in Mamba is making the state transition parameters input-dependent (selective SSM): the model learns which information to keep and which to forget, rather than using a fixed filter.

Analogy · stream vs. replay buffer

Attention is like a replay buffer: you store every past frame and can look back at any of them. The cost grows as the buffer fills. An SSM is more like a running stream consumer: you process each item and update a compact summary. The cost is constant per token regardless of context length. The trade-off: you can't look back at a specific frame, only query the current summary. For tasks requiring precise retrieval (exact quotes, long-range references), attention wins. For smooth, streaming tasks, SSMs are very fast.

What hasn't worked yet

Pure SSM models like early Mamba versions struggle on tasks that require precise recall of information from far back in the context: the type of thing attention handles trivially via direct lookup. Hybrid architectures (alternating SSM and attention layers) like Mamba-2 and Jamba try to get the best of both worlds, and this is active research territory. The jury is still out on whether SSMs can fully replace attention at scale.

Key paper
2023
Mamba: Linear-Time Sequence Modeling with Selective State Spaces Gu & Dao, Carnegie Mellon / Princeton

Demonstrated that a selective SSM could match Transformer quality on language modelling while scaling linearly (not quadratically) with sequence length. The most credible Transformer challenger to date. See reference [43] →

▸ How RoPE rotation works mathematically deep dive

RoPE applies a rotation matrix R(θ) to each pair of dimensions of Q and K, where θ = position × base_freq^(−2i/d) and i is the dimension index. This is exactly the encoding used in sinusoidal positional encoding, but applied via rotation rather than addition.

The dot product Q·K then computes: (R(m)q)·(R(n)k) = q·R(n−m)k, which depends only on the relative distance (n−m). This "relative position by construction" property is why RoPE generalizes better: you're never asking the model to recognize absolute positions it hasn't seen; it always sees relative distances which are learnable regardless of context length.

Long-context extensions (YaRN, NTK) work by rescaling the base frequency from the default 10,000 to a much higher value (e.g., 500,000), which stretches the rotation cycles so that the angular difference between positions that are far apart is still distinguishable.

Recap

  • GQA groups query heads to share fewer K/V projections: shrinks the KV cache 4–8× with minimal quality loss. Now standard in every major model family.
  • RoPE encodes relative position via rotation; extrapolates to unseen context lengths better than additive schemes. Long-context extensions (YaRN, NTK) push to 128K+ tokens.
  • MoE replaces the dense FFN with N expert networks and a learned router: massive parameter count, small per-token compute. The "47B" Mixtral runs like a 12B model.
  • SSMs (Mamba) replace attention with a selective linear recurrence, O(n) cost vs O(n²), but weaker on precise long-range recall. Hybrid models are the current frontier.
  • For production today: dense GQA + RoPE is the safe default; MoE if you need scale on a budget; SSMs worth watching.
