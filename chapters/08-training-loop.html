The Training Loop in Practice

The training loop is a distributed systems problem in disguise. Forward pass, backward pass, optimizer step: that's four lines of PyTorch. But fitting it on a single GPU is a memory-engineering challenge, and scaling to hundreds of GPUs is a full-blown sharding problem with all the complexity that implies.

Reading time · 22 min Prereqs · Plate 07 Code · PyTorch · accelerate

The training loop in five lines

Strip away all the engineering and the core training loop is surprisingly compact. Every gradient-based ML training framework (PyTorch, JAX, you name it) implements this same cycle:

python · minimal PyTorch training loop 
import torch
from torch.optim import AdamW
from torch.optim.lr_scheduler import CosineAnnealingLR
from transformers import AutoModelForCausalLM, AutoTokenizer

# --- Setup ---
device = "mps" if torch.backends.mps.is_available() else "cuda" if torch.cuda.is_available() else "cpu"
model_id = "HuggingFaceTB/SmolLM2-135M"
tok   = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id, torch_dtype=torch.float32).to(device)

optimizer = AdamW(model.parameters(), lr=3e-4, weight_decay=0.1)
scheduler = CosineAnnealingLR(optimizer, T_max=1000)

# --- Toy batch ---
text   = "The quick brown fox jumps over the lazy dog"
ids    = tok(text, return_tensors="pt").input_ids.to(device)
labels = ids.clone()

# --- The loop (this is all that matters) ---
for step in range(10):
    model.train()
    optimizer.zero_grad()                        # 1. clear old gradients

    outputs = model(ids, labels=labels)          # 2. forward pass → compute loss
    loss    = outputs.loss

    loss.backward()                              # 3. backward pass → compute gradients
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)  # gradient clip
    optimizer.step()                             # 4. update weights
    scheduler.step()                             # 5. adjust learning rate

    print(f"step {step:3d}  loss={loss.item():.4f}  lr={scheduler.get_last_lr()[0]:.2e}")

# Dummy dataset — replace with your DataLoader

ds     = TensorDataset(ids.repeat(32, 1))
loader = DataLoader(ds, batch_size=4)

model, optimizer, loader = accelerator.prepare(model, optimizer, loader)
scheduler = CosineAnnealingLR(optimizer, T_max=len(loader))

for step, (batch,) in enumerate(loader):
    optimizer.zero_grad()
    outputs = model(batch, labels=batch)
    loss = outputs.loss
    accelerator.backward(loss)                   # handles grad sync across GPUs
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()
    scheduler.step()
    if accelerator.is_main_process:
        print(f"step {step}  loss={loss.item():.4f}")

Five operations: zero gradients, forward, backward, clip, step. Everything else in this chapter is engineering to make those five operations run faster, use less memory, and stay numerically stable across trillions of steps.

Optimizers and learning-rate schedules

Nearly every LLM is trained with AdamW: Adam with weight decay decoupled from the gradient update. Recall from Plate 07 that Adam is expensive: per parameter, it stores two moment buffers (m, the first moment / exponential moving average of gradients, and v, the second moment / EMA of squared gradients) plus an fp32 master copy of the weights. That's 4 × the memory of the weights themselves, just for the optimizer state. For a 7 B model in bf16, the weights are ~14 GB; the AdamW state is an additional ~56 GB. This is the single biggest memory cost in training.

The learning rate is the most consequential hyperparameter in training, more than batch size, dropout, or most architectural choices. The standard schedule has two phases:

  • Linear warmup. Start with a very small learning rate and linearly ramp it to the peak value over a few hundred to a few thousand steps. Skipping warmup on large models causes early training instability because the large initial gradients can corrupt the random initialization irreversibly.
  • Cosine decay. After warmup, decay the learning rate following a cosine curve down to ~10% of peak by the end of training. Cosine decay is preferred over linear decay because it slows down naturally near the end, letting the model settle into a good local minimum rather than stopping abruptly.
Gotcha · the learning rate is the most sensitive knob

Too high: loss spikes or diverges, gradients explode, the model never converges. Too low: training is unnecessarily slow; you waste compute. The sweet spot is empirically derived; most modern runs use a peak LR of 1e-4 to 3e-4 for 7B-scale models, often found by running a short LR sweep at 1% of the full run budget. Don't copy hyperparameters from a paper without checking whether the model size and batch size match yours; the optimal LR scales with both.

Fitting it in memory: mixed precision, grad accumulation, checkpointing

A GPU has fast on-chip SRAM (the registers, L1 cache) and much larger but slower HBM (high-bandwidth memory, the VRAM you buy). OOM (out-of-memory) errors are the bane of training engineers. The toolkit for fighting them:

Mixed precision (bf16/fp16). Compute in 16-bit floats; keep the optimizer's master weights in fp32. The forward and backward passes run in bf16, which is 2× faster on Ampere and Hopper GPUs (the Tensor Cores are designed for it) and uses half the memory for activations. The fp32 master weights ensure that the small gradient updates don't round to zero. BF16 is strongly preferred over FP16 for training: its wider dynamic range matches fp32 and avoids the overflow/underflow issues that require loss scaling under FP16.

Analogy · mixed precision is like a calculator

You do the intermediate arithmetic on a basic calculator (fast, 8-significant-figures), but write down results in a precise lab notebook (32 digits). The final totals are as accurate as the notebook; you just didn't need high precision for every intermediate step. That's bf16 activations + fp32 master weights in a nutshell.

Gradient accumulation. Large batch sizes reduce gradient noise and improve training stability, but doubling the batch size doubles activation memory. Gradient accumulation simulates a large batch by running multiple small forward-backward passes without calling optimizer.step(), then summing the gradients before the update. With accumulation steps = 8, you get the effective batch size of 8× your physical batch at the cost of 8× the time, but no extra memory. This is standard practice on memory-constrained hardware.

Gradient clipping. Cap gradient norms at a threshold (typically 1.0). When a batch contains anomalous text, the loss spikes, gradients explode, and without clipping the optimizer takes a huge wild step that can take hours of training to recover from. Clipping is a cheap guard against catastrophic steps. Monitor the gradient norm over training. If it's consistently hitting the clip threshold, your LR may be too high.

Activation checkpointing (gradient checkpointing). During the forward pass, intermediate activations are normally stored for the backward pass. For a 32-layer model with 4k context, that's enormous. Gradient checkpointing discards intermediate activations during the forward pass and recomputes them on-the-fly during the backward pass. The trade-off: ~33% slower backward pass, but activation memory drops dramatically. Standard for any model that doesn't fit otherwise.

When one GPU isn't enough: data, tensor, pipeline parallelism

A single A100 80 GB can hold roughly a 40B-parameter model in fp16, with nothing left for activations or optimizer state. Pretraining at 7B or above almost certainly requires multiple GPUs. There are three orthogonal ways to split the work:

From a software-eng POV · distributed training = sharding state across workers

You already know this problem. In distributed databases, you shard either the data (rows) or the schema (columns/tables) across nodes. In distributed training, you shard either the data (different batches per GPU), the model's layers (different layers per GPU: pipeline parallelism), or the tensors within a layer (split weight matrices across GPUs: tensor parallelism). The collective communication primitives (all-reduce, all-gather, scatter) are exactly the reduce/broadcast operations you'd see in a map-reduce pipeline. ZeRO is the optimizer-state equivalent of a sharded database, which we'll get to next.

Data parallelism (DP). The simplest strategy: each GPU holds a full copy of the model and processes a different mini-batch. After each forward-backward, gradients are all-reduced (summed and averaged) across GPUs before the optimizer step. PyTorch's DistributedDataParallel (DDP) implements this and is the default for anything that fits on one GPU. Scales linearly in throughput with GPU count. The limitation: you still need the full model, optimizer state, and gradients on every GPU.

Tensor parallelism (TP). Split individual weight matrices across GPUs. For a matrix multiply Y = XW, column-split W across k GPUs: each GPU computes a slice of Y, and the slices are assembled via all-gather. Megatron-LM made this famous. TP requires high-bandwidth inter-GPU links (NVLink) because it needs to synchronise at every layer. Within a node: great. Across nodes over InfiniBand: painful.

Pipeline parallelism (PP). Assign sequential layer groups to different GPUs: GPU 0 handles layers 0–7, GPU 1 layers 8–15, and so on. Input flows through the pipeline stage by stage. Requires micro-batching to keep all stages busy (otherwise GPUs stall waiting for the next batch). PP has low communication volume (just layer-boundary activations) but introduces "pipeline bubbles" where GPUs idle at the start and end of each batch.

Real large-scale runs combine all three: 3D parallelism. A 1,000-GPU run might use TP = 8 (within-node), PP = 8 (cross-node), and DP = 16 (data replicas). Coordinating this is a significant infrastructure problem, one reason large-scale training is hard to replicate.

FSDP and ZeRO: sharding the model itself

For most engineers who aren't running frontier-scale pretraining, the more practical tool is FSDP (Fully Sharded Data Parallel, PyTorch native) or DeepSpeed ZeRO (Zero Redundancy Optimizer). Both implement the same core insight: in standard DDP, every GPU holds a full copy of the model, gradients, and optimizer state; with N GPUs, that's N × the total memory cost. Why replicate? Shard it.

ZeRO / FSDP shard across the GPU ranks:

  • Stage 1 (ZeRO-1): shard optimizer state. Each GPU holds 1/N of the Adam m and v buffers. Still redundant weights and gradients.
  • Stage 2 (ZeRO-2): also shard gradients. After the backward pass, each GPU only holds its slice of the gradients.
  • Stage 3 / FSDP: also shard the parameters themselves. During the forward and backward passes, each layer's weights are all-gathered from the shards as needed, then freed. The communication overhead is one all-gather per layer per forward and one reduce-scatter per backward, but peak memory drops to 1/N of the total model state.
From a software-eng POV · ZeRO = sharding the optimizer-state database

Think of the AdamW optimizer state as a key-value store where the key is "parameter index" and the value is (weight, first moment, second moment). DDP replicates this database on every GPU, a wasteful full copy. ZeRO applies horizontal sharding: each GPU owns a range of parameter indices and holds only those rows. During the optimizer step, each GPU updates its shard locally: no cross-GPU communication for the update itself. The all-gather before the forward pass is the equivalent of a distributed join: assemble the full parameter tensor from its shards. It's map-reduce for the optimizer.

# accelerate config (or write ~/.cache/huggingface/accelerate/default_config.yaml directly)

cat > fsdp_config.yaml <<'EOF'

distributed_type: FSDP

  fsdp_auto_wrap_policy: TRANSFORMER_BASED_WRAP

EOF

Monitoring a run

A pretraining run lasting weeks is a long-running stateful process, exactly the kind of thing engineers instrument obsessively in production systems. The same instincts apply here:

  • Training loss curve. Should decrease smoothly. A spike followed by recovery usually indicates a bad data batch. A spike with no recovery is a divergence: abort, restore last checkpoint, and investigate before restarting (usually a LR too high or an inf/NaN in the data).
  • Gradient norm. Log the gradient norm every step. Should be roughly stable after warmup. Consistently high (always hitting clip threshold) → LR too high or data issues. Steadily decreasing toward zero → vanishing gradients or LR too low.
  • GPU utilisation / MFU. Model FLOP utilisation (MFU) is the fraction of peak GPU FLOP/s you're actually achieving. Good runs target 40–60% MFU. Below 30% suggests a data pipeline bottleneck (CPU data loading is starving the GPU), suboptimal kernel fusion, or excessive communication overhead in the distributed setup.
  • Validation loss. Evaluate on a held-out slice every N steps. If training loss drops but validation loss stagnates or rises, you're overfitting: either reduce the number of epochs or add regularization. For large pretraining runs you typically see only one epoch over the data, so overfitting is rare; contamination is a bigger concern.
  • Checkpoint cadence. Save a full checkpoint every few thousand steps. On a multi-week run, losing the last 24 hours of training to a hardware fault (it will happen) is painful; losing 3 days is a disaster. Most teams also keep the best N checkpoints by validation loss and delete the rest.
Gotcha · OOM debugging is an art

Out-of-memory errors in CUDA are notoriously unhelpful. Common causes and fixes: (1) activations too large → enable gradient checkpointing; (2) batch size too large → reduce, or use gradient accumulation; (3) optimizer state not sharded → switch to FSDP or DeepSpeed ZeRO-2; (4) mixed precision not enabled → switch to bf16; (5) KV cache being allocated unnecessarily during training (it's for inference only; make sure use_cache=False is set in the model config for training). Profile with torch.cuda.memory_summary() and PyTorch's memory snapshot tool before reaching for more GPUs. You might be leaving 20 GB on the table.

For a 7B model trained in bf16 with AdamW, per-parameter costs are:

ComponentDtypeBytes/param
Model weightsbf162
Gradientsfp324

7B × 18 bytes ≈ 126 GB before activations. On 4 × A100 80 GB GPUs (320 GB aggregate) with FSDP Stage 3, each GPU holds 126 / 4 ≈ 31.5 GB of model state, leaving ~48 GB per GPU for activations, enough for reasonable context lengths and batch sizes. This is the back-of-envelope calculation to size your GPU cluster before starting a run.

Recap

  • The training loop is: zero gradients → forward pass → backward pass → gradient clip → optimizer step → LR scheduler step. Five lines; everything else is engineering.
  • AdamW costs ~16–18 bytes per parameter: weights + gradients + two moment buffers + fp32 master copy. Optimizer state dominates memory.
  • The learning rate is the most sensitive hyperparameter. Use linear warmup followed by cosine decay; tune peak LR carefully.
  • Mixed precision (bf16) halves activation memory and doubles compute speed. Gradient accumulation simulates large batches without extra memory. Gradient checkpointing trades 33% compute for large activation memory savings.
  • Data parallelism replicates the model across GPUs, syncing gradients via all-reduce. Tensor parallelism splits matrices within a layer. Pipeline parallelism splits layers across GPUs.
  • FSDP / ZeRO shard the model weights, gradients, and optimizer state across GPUs; think of it as horizontally sharding the optimizer-state key-value store. Stage 3 reduces per-GPU memory by ~N× for N GPUs.
  • Monitor: training loss smoothness, gradient norm stability, MFU (>40% is healthy), validation loss, and checkpoint cadence.
