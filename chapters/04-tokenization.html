Tokenization & Embeddings

A model can't read. It does arithmetic on numbers. Tokenization is the boundary layer that turns your text into integers, and it quietly explains why models cost what they cost, why they're bad at spelling, and why your French prompt is more expensive than your English one.

Reading time · 14 min Prereqs · Plate 01 Code · transformers · tiktoken Widget · Tokenizer playground

Computers don't read; they multiply

Everything a language model does (the billions of parameters, the attention of Plate 05) is matrix multiplication over floating-point numbers. But your prompt is a string. Something has to translate. That something is the tokenizer: a deterministic, learned function that maps text to a sequence of integer IDs and back again.

From a software-eng POV

The tokenizer is a lexer. It scans a character stream and emits a sequence of symbols (tokens) drawn from a fixed vocabulary, exactly like tokenizing source code before parsing. The difference: this lexer's rules were learned from a corpus, not written by hand, and each token maps to an integer ID via a lookup table.

Why subwords won: a tale of two bad options

The obvious approaches both failed, and understanding why is the whole story.

What didn't work · character-level

Give the model one token per character. The vocabulary is tiny (~100 symbols) and nothing is ever "unknown." But sequences become enormous ("tokenization" is 12 tokens instead of 1), and the model wastes capacity relearning that t-h-e spells "the." Attention is quadratic in sequence length (Plate 05), so this is ruinously expensive.

What didn't work · word-level

Give the model one token per word. Sequences are short, but the vocabulary explodes (millions of words, plurals, typos, names) and any word not seen in training is a dreaded <UNK>: out-of-vocabulary, information lost. You can't enumerate a language.

What won · subword tokenization (BPE)

Split text into subword pieces. Common words stay whole ("the", "model"); rare words break into parts ("tokeni" + "zation"); truly novel strings fall back to bytes. Vocabulary stays bounded (~30k–130k), sequences stay short, and nothing is ever out-of-vocabulary. Introduced to NLP via Byte-Pair Encoding in 2016 and refined by SentencePiece, it's still how essentially every modern LLM tokenizes.

How byte-pair encoding actually works

BPE is almost embarrassingly simple: it's a compression algorithm. Start with text as a sequence of bytes/characters. Find the most frequent adjacent pair, merge it into a new symbol, and record the merge. Repeat thousands of times. The learned list of merges is the tokenizer.

Analogy · building a phrasebook by frequency

Imagine compressing a book by repeatedly finding the two-symbol sequence that occurs most and giving it a shorthand. First "e"+space, then "th", then "the", then "ing"… After enough passes your shorthand covers most common chunks. BPE is that, run to a target vocabulary size. The vocabulary self-organizes around what's frequent.

python · inspect a tokenizer 
# pip install transformers
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("gpt2")     # any model id works
text = "Fine-tuning with LoRA is surprisingly cheap."

ids = tok(text)["input_ids"]
print(ids)                                      # the integers the model sees
print(tok.convert_ids_to_tokens(ids))           # the human-readable pieces ('Ġ' = leading space)
print(len(ids), "tokens for", len(text), "chars")
# pip install tiktoken
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")       # GPT-3.5/4 family
text = "Fine-tuning with LoRA is surprisingly cheap."

ids = enc.encode(text)
print(ids, len(ids))
print([enc.decode([i]) for i in ids])            # decode each id back to its piece

Tokens have consequences

Tokenization isn't a boring preprocessing detail; it leaks into everything:

  • Cost & limits are per-token. APIs bill per token; the context window is measured in tokens. "~4 characters per token" for English is the rule of thumb, so ~750 words ≈ 1,000 tokens.
  • Non-English text costs more. Tokenizers are trained mostly on English, so other languages (and code, and emoji) fragment into more tokens (sometimes 2–3×), making them slower and pricier for the same meaning.
  • Models "can't spell" or count letters because they never see letters; they see "Ġstrawberry" as one opaque token. The famous "how many r's in strawberry" failure is a tokenization artifact, not a reasoning one.
  • Arithmetic is hard partly because numbers tokenize inconsistently ("1234" might be one token, "1235" two). Newer tokenizers split digits individually to help.

From IDs to meaning: the embedding table

A token ID like 5689 is just an index; it carries no meaning (5689 isn't "near" 5690 in any useful sense). The first thing the model does is look that ID up in an embedding table: a big learnable matrix with one row (a vector of, say, 4096 floats) per vocabulary entry. That vector is the token's embedding: its learned meaning.

Analogy · a hashmap to a coordinate

The embedding table is a dict[int, float[]]: token ID → point in high-dimensional space. Crucially, the points are arranged so that distance encodes similarity: "king" sits near "queen," "Paris" near "London." Training nudges these coordinates until the geometry of the space mirrors the structure of language.

This idea predates LLMs. The 2013 word2vec work showed that learned word vectors capture analogies as arithmetic: the vector for "king" minus "man" plus "woman" lands near "queen." The neural language model that started it all had introduced learned embeddings a decade earlier. In a Transformer, the embedding table is learned jointly with everything else, and the same matrix is often reused at the output to turn final vectors back into logits over the vocabulary (Plate 06).

▸ Token embeddings vs. "an embedding model" deep dive

Don't confuse two uses of the word. The embedding table inside an LLM gives each token a vector. An embedding model (Plate 17) is a separate model that maps an entire sentence or document to one vector for semantic search and RAG. Related idea (meaning as geometry), different jobs: one is a lookup row, the other is a whole forward pass producing a single pooled vector.

Recap

  • Models operate on integers; the tokenizer is the learned lexer that converts text ↔ token IDs.
  • Subword (BPE) tokenization beat character- and word-level by keeping the vocabulary bounded and never hitting "unknown."
  • Tokens (not characters or words) are the unit of cost, context, and many model quirks (spelling, math, non-English overhead).
  • The embedding table maps each token ID to a learned vector where distance ≈ meaning; this is where the model's understanding begins.

With tokens embedded as vectors, we're ready for the engine that mixes them together. Onward to Plate 05, the Transformer.
