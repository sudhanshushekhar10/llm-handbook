Quantization & GGUF

A 70B model in full precision is ~140 GB; it won't fit anywhere cheap. Quantization stores the weights at lower numeric precision, trading a sliver of quality for a 4× shrink. It's the reason a capable LLM now runs on your laptop.

Reading time · 17 min Prereqs · Plate 06 Code · bitsandbytes · llama.cpp · Ollama Widget · Quantization explorer

The memory wall

A model's weights are numbers, and numbers take space. In fp32 (32-bit float) each parameter is 4 bytes; in fp16, 2 bytes. Inference is almost entirely memory-bandwidth-bound (the GPU spends its time hauling weights from memory, not doing math), so smaller weights mean both a smaller footprint and faster generation. That's the double win.

Analogy · lowering the bit-depth of an image

Quantization is lossy compression for weights. Like saving a photo as an 8-bit PNG instead of 16-bit, or an MP3 instead of WAV: you map a fine-grained range onto a coarser set of levels. Done well, the eye (or here, the model's output) barely notices. Done too aggressively, artifacts appear.

Gotcha · fp16 vs bf16 are not the same 16 bits

Both are 2 bytes, but they split those bits differently. fp16 spends more bits on precision and fewer on range; bf16 (brain-float) keeps fp32's full range with less precision. Training likes bf16's range (gradients vary wildly and rarely overflow). It's a classic engineering trade: dynamic range vs. granularity.

Play with the tradeoff. Notice the cliff: from fp16 down to ~4 bits, quality holds remarkably well; below that it falls off fast while the size barely improves. There's a clear sweet spot.

Doing it well: outliers and smart quantization

Naive rounding to int4 wrecks quality, because a few weights have outsized magnitudes ("outliers") that, when squashed, distort everything. The research that made low-bit LLMs practical is mostly about handling those outliers:

  • LLM.int8(): keep the rare outlier dimensions in 16-bit and quantize the rest to int8. Near-lossless 8-bit, and the foundation of the bitsandbytes library.
  • GPTQ: a one-shot, layer-by-layer method that compensates for the error each quantization step introduces. Fast 3–4 bit weights for GPU inference.
  • AWQ ("activation-aware"): identify the small fraction of weights that matter most (judged by activation magnitudes) and protect them. Strong 4-bit quality and speed.
From a software-eng POV

These are codecs. Just as H.264 and AV1 are different algorithms for compressing the same video with different quality/speed/size tradeoffs, GPTQ, AWQ, and the GGUF k-quants are different weight codecs. You pick one based on your runtime (GPU vs CPU) and your tolerance for the size/quality knob.

GGUF, llama.cpp, and the laptop revolution

On the CPU/consumer side, one project changed the game: llama.cpp, a C/C++ engine that runs quantized models efficiently on ordinary hardware, including Apple Silicon via Metal. Its model format is GGUF: a single file bundling the quantized weights, the tokenizer, and metadata.

GGUF's "k-quant" levels are the names you'll see everywhere: Q4_K_M, Q5_K_M, Q8_0, Q2_K. The number is roughly the bits per weight; K means a smarter mixed-precision scheme; _M / _S are medium/small variants. Q4_K_M is the community default, about a quarter of the size of fp16 with quality almost indistinguishable in practice. Ollama wraps all of this so ollama run llama3.2 just works.

Hands-on: run a quantized model

python · load a quantized model 
# Easiest path on a Mac — Ollama pulls a Q4_K_M GGUF automatically:
#   brew install ollama
#   ollama run llama3.2            # chat right in the terminal

# Or drive a GGUF directly from Python (CPU / Apple Metal):
# pip install llama-cpp-python
from llama_cpp import Llama

llm = Llama.from_pretrained(
    repo_id="bartowski/Llama-3.2-3B-Instruct-GGUF",
    filename="*Q4_K_M.gguf",          # the sweet-spot quant
    n_ctx=4096,
)
out = llm("Q: Why is the sky blue?\nA:", max_tokens=64)
print(out["choices"][0]["text"].strip())
# pip install torch transformers bitsandbytes accelerate
# Load an 8B model in 4-bit on an NVIDIA GPU — ~6 GB instead of ~16 GB.
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

bnb = BitsAndBytesConfig(
    load_in_4bit=True, bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
)
name = "meta-llama/Llama-3.1-8B-Instruct"
tok   = AutoTokenizer.from_pretrained(name)
model = AutoModelForCausalLM.from_pretrained(name, quantization_config=bnb, device_map="auto")

msgs = [{"role": "user", "content": "Why is the sky blue?"}]
inputs = tok.apply_chat_template(msgs, add_generation_prompt=True, return_tensors="pt").to(model.device)
print(tok.decode(model.generate(inputs, max_new_tokens=64)[0], skip_special_tokens=True))
output
The sky is blue because of Rayleigh scattering: air molecules scatter
shorter (blue) wavelengths of sunlight more than longer (red) ones...
# A 3B Q4_K_M model produces this on a laptop, using ~2 GB of RAM.
What breaks, and when

Below ~3 bits (e.g. Q2_K), quality degrades sharply: the model gets repetitive, loses instruction following, and hallucinates more. And quantization compounds with everything else; a heavily quantized and aggressively pruned and tiny model is often worse than a smaller model at a higher precision. Prefer a smaller model at Q4_K_M over a bigger one at Q2.

Recap

  • Quantization stores weights at lower precision: a lossy codec that shrinks models ~4× and speeds up memory-bound inference.
  • fp16/bf16 → int8 → int4: quality holds well down to ~4 bits, then falls off a cliff.
  • Good methods (LLM.int8, GPTQ, AWQ) work by protecting outlier weights.
  • GGUF + llama.cpp (and Ollama) brought quantized models to laptops; Q4_K_M is the default sweet spot.
  • Rule of thumb: a smaller model at Q4_K_M usually beats a larger model crushed to Q2.
