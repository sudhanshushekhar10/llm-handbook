Inference & Serving

An LLM talking to a single user on idle hardware is a solved problem. An LLM fielding thousands of concurrent requests with tight latency SLOs is a hard systems problem — one that looks surprisingly familiar once you understand the two phases of generation.

Reading time · 20 min Prereqs · Plates 06, 13 Code · Ollama · vLLM · OpenAI-compatible client

Two phases: prefill and decode

Every time you send a prompt to an inference server, generation splits into two very different workloads.

Prefill is the first pass: the entire prompt — say, 500 tokens of system instructions plus your user message — is fed through the model in one parallel forward pass. Because all prompt tokens are available simultaneously, the GPU can compute attention across the full sequence in one shot, just like training. This is compute-bound: the GPU's tensor cores are the bottleneck.

Decode is everything after: generating the response one autoregressive token at a time. Each new token depends on all previous ones, so the model does a separate forward pass per token. There's no way to parallelise within a sequence. Worse, the model loads nearly all of its weights from GPU memory for every single token — and generates only a handful of floating-point results in return. This is memory-bandwidth-bound: the GPU spends most of its time waiting for memory, not computing.

From a software-eng POV

Prefill is like a batch job: lots of data, fully parallelisable, predictable duration. Decode is like a tight request/response loop: one result per round-trip, dominated by I/O latency. The serving stack has to handle both modes efficiently at the same time — with different batching strategies for each.

The implication for metrics is immediate. Time to first token (TTFT) is determined almost entirely by prefill latency — how quickly the server can process your prompt. Tokens per second during decode is determined by how fast weights can be streamed from GPU HBM. A GPU with more memory bandwidth (not just raw FLOPs) produces faster generation on long outputs.

The KV cache: memoization for attention

Consider what the decode phase actually computes. At each step, attention asks: given the current query token, what do all the previous keys and values say? But those previous keys and values haven't changed — they were computed during prefill and earlier decode steps. Recomputing them from scratch on every single decode step would be catastrophically wasteful.

The KV cache is the solution: store the key and value tensors for every past token in GPU memory and reuse them. At step t, only the single new query needs to attend over the full cached key-value sequence. This turns an O(n²) recomputation into an O(n) lookup.

Analogy · memoization

If you've ever written a recursive Fibonacci function and then added a cache to avoid recomputing sub-results, you've done exactly this. The KV cache is @lru_cache for the attention layer: the first forward pass is expensive (like computing fib(30) from scratch), and every subsequent decode step is cheap (like returning the cached result). The tradeoff is the same too — you trade memory for time.

The catch: KV cache memory is large and grows with every token generated. For a single request, the cache holds two tensors (K and V) per layer, per head, per token. A 70B model with 80 layers, 8 KV heads (GQA), and a head dimension of 128, generating 2,048 tokens in fp16, consumes roughly: 

2 (K+V) × 80 layers × 8 heads × 128 dim × 2,048 tokens × 2 bytes
≈ 1.3 GB for a single request

Multiply by 50 concurrent requests and you've consumed 65 GB — which may exceed the space available for weights. This is the central resource-management problem of LLM serving.

Gotcha · KV cache memory explodes at long context

KV cache size scales linearly with context window length and with the number of concurrent requests. A 128K-context model serving 20 parallel users can need hundreds of gigabytes of KV cache — more than the weights themselves. This is exactly why GQA (Grouped Query Attention, Plate 15) was invented: sharing key and value heads across groups of query heads cuts the KV cache by 4–8× at no meaningful quality cost. And it's why quantization (Plate 13) of the KV cache itself is increasingly common — storing K/V in int8 instead of fp16 halves cache memory.

Batching: keeping the GPU busy

A GPU running decode for a single request is nearly idle — it loads gigabytes of weights to compute a handful of values per token. The utilisation numbers are dismal: often below 5%. The fix is batching — processing multiple requests through the same forward pass so those weight loads are amortised across many concurrent decode steps.

Static batching (the naive approach) groups requests at arrival time and runs them together until all finish. The problem is padding: the shortest sequence in the batch must wait for the longest to complete. A batch of 32 requests where one is generating a 2,000-token essay holds the other 31 hostage for its entire duration. GPU memory fills with KV cache for that one long sequence, and new requests queue behind it.

Continuous batching (also called in-flight or iteration-level batching) solves this. Instead of locking a batch for the duration of all sequences, the scheduler makes a new batching decision at every single decode step. As soon as one sequence finishes generating and emits its end-of-sequence token, a waiting request is immediately slotted in to fill that slot. The GPU batch is therefore always full, and no single long request can monopolise capacity.

Analogy · work-stealing thread pool / connection pool

Static batching is like a fixed-size thread pool where every thread grabs a task and holds it until done, refusing to take a new one even if its task finishes early. Continuous batching is like a work-stealing scheduler or a database connection pool: the moment a slot frees up, it's reassigned to the next waiter. The result is the same high-throughput behaviour you'd expect from a well-tuned HTTP server under heterogeneous request durations.

PagedAttention and continuous batching (vLLM)

Continuous batching is only half the story. The second problem is KV cache fragmentation. Because requests vary in length and arrive at different times, a naively managed contiguous KV cache ends up with fragmented free blocks — similar to heap fragmentation in a long-running process. You might have 40 GB free in theory but be unable to allocate a contiguous 2 GB block for a new long request.

PagedAttention, introduced by the vLLM team, borrows the operating system's solution to exactly this problem: virtual memory paging. Instead of pre-allocating a contiguous chunk for each request, the KV cache is divided into fixed-size blocks (pages). Each request is assigned a page table mapping its logical token positions to physical GPU memory blocks. Pages are allocated on demand and freed immediately when a request completes. Near zero waste, zero fragmentation, and the ability to share pages between requests that share a common prefix (system prompt caching).

vLLM combines PagedAttention with continuous batching into a single serving engine. It exposes an OpenAI-compatible REST API, making it a drop-in replacement for most applications. On GPU clusters, vLLM consistently achieves 3–10× higher throughput than naive serving because the GPU stays saturated — no wasted cycles waiting on long-tail requests or fragmented memory.

FlashAttention and speculative decoding

Two more techniques appear in nearly every serious serving stack; it's worth knowing what they actually do.

FlashAttention

Standard attention computes the full N×N attention matrix and materialises it in GPU HBM (high-bandwidth memory). For a 4K-token sequence, that's a 4,096 × 4,096 matrix of floats — 64 MB per layer, and it must be written and read twice (once to compute softmax, once to weight the values). FlashAttention is an IO-aware rewrite of the attention kernel: it never materialises the full matrix. Instead, it tiles the computation into SRAM-sized chunks that fit entirely in the GPU's on-chip cache, fusing the softmax and the weighted sum into a single kernel pass. The result is mathematically exact (not approximate) and roughly 2–4× faster, with memory usage that is O(N) instead of O(N²). FlashAttention is now the default in every serious framework.

From a software-eng POV

FlashAttention is a kernel-level cache-oblivious algorithm — the LLM equivalent of blocking a matrix multiply to fit in L3 cache. It doesn't change the math at all; it changes the memory access pattern. A GPU's SRAM is 1,000× faster than HBM, so eliminating the round-trips to HBM is the entire win. Same output. Radically different I/O profile.

Speculative decoding

Speculative decoding attacks the fundamental bottleneck of autoregressive generation from a different angle. Observation: the large model (the "verifier") is doing one token per step, but a much smaller draft model (say, 1B instead of 70B) can generate several tokens almost instantly. Those draft tokens are probably right most of the time — simple continuations are predictable.

The trick: run the draft model for k steps to produce k candidate tokens, then feed all k+1 positions (prompt + drafts) to the large model in a single parallel forward pass for verification. If the large model agrees on position i, accept it; when it first disagrees, reject from that point and substitute the large model's token. On easy, predictable text (code comments, bullet lists, boilerplate), acceptance rates of 70–90% are typical — meaning you get 3–5 effective tokens for the cost of one large-model forward pass. Throughput goes up; correctness is guaranteed by construction.

The cost is running a second model and the coordination overhead. Speculative decoding is most effective when the distribution of outputs is narrow (the draft model guesses right often), less effective on creative or stochastic outputs.

Choosing a serving stack

The right engine depends on your hardware, scale, and operational preferences. Here is the practical breakdown:

python · OpenAI-compatible client 
# pip install openai
from openai import OpenAI

# Point at your local Ollama server (or swap base_url for vLLM's address):
client = OpenAI(
    base_url="http://localhost:11434/v1",  # Ollama
    # base_url="http://localhost:8000/v1",  # vLLM
    api_key="ollama",                       # placeholder; Ollama ignores the key
)

response = client.chat.completions.create(
    model="llama3.2",      # or "meta-llama/Llama-3.1-8B-Instruct" for vLLM
    messages=[{"role": "user", "content": "Explain the KV cache in one paragraph."}],
    max_tokens=200,
    stream=True,           # stream tokens as they are decoded
)

for chunk in response:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)

The metrics that matter

A serving system has a latency dimension and a throughput dimension, and they pull in opposite directions. Here are the three numbers every operator watches:

MetricDefinitionWhat drives itTypical target
TTFT (Time to First Token) Wall-clock time from request receipt to the first token in the response stream Prefill latency; prompt length; server queue depth <200 ms for interactive; <2 s for batch
TPOT (Time per Output Token) Average latency per generated token after the first GPU memory bandwidth; batch size; model size <50 ms/token ≈ 20 tok/s for readable streaming
Throughput (tok/s, system-wide) Total tokens generated per second across all concurrent requests Batch size; GPU utilisation; continuous batching efficiency Maximise for batch workloads; trade off against TTFT for interactive

The fundamental tension: increasing batch size improves GPU utilisation (throughput goes up) but also increases queuing delay for new requests (TTFT goes up). A well-tuned continuous batching engine finds the operating point that saturates the GPU without letting queue depths grow unbounded — exactly the problem any async backend engineer has solved in a different context.

For monitoring, instrument your serving stack to emit per-request TTFT, TPOT, and prompt/completion token counts. Aggregate them at p50/p95/p99. A spike in p95 TTFT usually means the queue is backing up (add capacity or raise batch priority). A spike in TPOT usually means the GPU is undersaturated (bigger batch) or memory-bandwidth is exhausted (quantize the KV cache or switch to a GQA model).

Recap

  • Generation has two phases: prefill (the whole prompt, compute-bound, fast) and decode (one token at a time, memory-bandwidth-bound, the real bottleneck).
  • The KV cache is memoization for attention: past keys and values are stored so only the new query needs to be computed. It dominates GPU memory and grows with context length and batch size.
  • Continuous batching refills the decode batch on every step as sequences finish, keeping GPU utilisation near 100%.
  • PagedAttention (vLLM) applies virtual memory paging to the KV cache, eliminating fragmentation and enabling prefix sharing.
  • FlashAttention is an IO-aware kernel that avoids materialising the N×N attention matrix, giving exact attention at O(N) memory.
  • Speculative decoding uses a cheap draft model to propose tokens in parallel; the large model verifies them in one forward pass, giving 3–5× effective throughput on predictable text.
  • Stack choice: vLLM or TGI for GPU production; SGLang for prefix-heavy workloads; Ollama/llama.cpp for local development.
  • Watch TTFT, TPOT, and system throughput. Latency and throughput trade off — continuous batching is what keeps both acceptable simultaneously.

The serving layer is now efficient. The next question is the shape of the model itself — giant parameter counts, smarter attention, and architectures that aren't transformers at all. Next: advanced architectures.
