Data for Fine-tuning

Training algorithms are the same for everyone. The dataset is the moat. A small, clean, well-formatted dataset consistently beats a large, noisy one — and getting the format wrong (especially the chat template) can silently waste weeks of GPU time. This plate is the practical guide to building training data that actually works.

Reading time · 18 min Prereqs · Plates 09, 10 Code · pydantic · transformers

Formats: instruction, chat, and preference data

Before you write a single training example, you need to decide on your data format. There are three standard shapes, each suited to a different training objective.

Instruction format (Alpaca-style). The simplest shape: each example is a flat string containing an instruction and a response, separated by a fixed delimiter. Popularized by Stanford Alpaca, this format is easy to produce and works well for single-turn tasks (classification, extraction, transformation).

jsonl · instruction format (Alpaca)
{"instruction": "Summarize this in one sentence.", "input": "The Eiffel Tower...", "output": "The Eiffel Tower is a 330-meter iron lattice tower built in Paris for the 1889 World's Fair."}
{"instruction": "Convert this date to ISO 8601.", "input": "June 14, 2026", "output": "2026-06-14"}

Chat format (messages with roles). Multi-turn conversations represented as a list of role-tagged messages. This is what most modern chat models expect — and it is the format you must use if you are fine-tuning a chat/instruct variant. The roles are typically system, user, and assistant.

jsonl · chat format (messages list)
{"messages": [
  {"role": "system", "content": "You are a concise technical writer."},
  {"role": "user", "content": "What is a B-tree?"},
  {"role": "assistant", "content": "A B-tree is a self-balancing search tree where each node can have multiple children, optimized for read/write on disk by keeping node size equal to a disk block."}
]}
{"messages": [
  {"role": "user", "content": "Write a Python function to binary search a sorted list."},
  {"role": "assistant", "content": "def binary_search(arr, target):\n    lo, hi = 0, len(arr)-1\n    while lo <= hi:\n        mid = (lo+hi)//2\n        if arr[mid] == target: return mid\n        elif arr[mid] < target: lo = mid+1\n        else: hi = mid-1\n    return -1"}
]}

Preference format (chosen/rejected pairs). Used for DPO and RLHF (Plate 11). Each example includes a prompt plus two responses: one the model should learn to prefer (chosen) and one it should learn to avoid (rejected). The contrast is the training signal.

jsonl · preference format (DPO)
{"prompt": "What is 17 × 6?",
 "chosen": "102.",
 "rejected": "Around 100, give or take."}
{"prompt": "Explain why quicksort is not stable.",
 "chosen": "Quicksort is not stable because the partition step can swap equal elements past each other, changing their relative order.",
 "rejected": "Quicksort uses a pivot to sort. It is very fast on average — O(n log n) — but in the worst case it degrades to O(n²)."}

Chat templates and special tokens (where fine-tunes go to die)

Here is the single most common cause of silent fine-tuning failure: wrong chat template.

When a model is instruction-tuned, it learns to respond after seeing a specific sequence of special tokens that delimit turns — think of them as HTTP headers for conversation. Llama-3 uses <|start_header_id|>user<|end_header_id|>. Mistral uses [INST] / [/INST]. Phi-3 uses <|user|> / <|assistant|>. These are not interchangeable. If you format your training data using Mistral's template but fine-tune Llama-3, the model sees garbled input — the "headers" land in the wrong places and the model has no idea where one turn ends and the next begins.

The fix is to use tokenizer.apply_chat_template(), which reads the template baked into the tokenizer config and applies it automatically. Always use it. Never format manually.

Gotcha · wrong chat template silently ruins fine-tunes

The training loss will still fall. The model will still produce text. You will not see an error. You will only discover the problem when you run inference and the model ignores your instructions, hallucinates role names, or refuses to stop generating. Chat template mismatches are often diagnosed weeks after a fine-tune completes. Use tokenizer.apply_chat_template() and then print a decoded example to verify before training.

python · apply_chat_template correctly 
from transformers import AutoTokenizer

model_id = "mistralai/Mistral-7B-Instruct-v0.3"
tokenizer = AutoTokenizer.from_pretrained(model_id)

# Raw messages list — model-agnostic
messages = [
    {"role": "system", "content": "You are a helpful coding assistant."},
    {"role": "user", "content": "Write a Python hello world."},
    {"role": "assistant", "content": 'print("Hello, world!")'},
]

# apply_chat_template reads the tokenizer's built-in jinja template
# add_generation_prompt=False because we're formatting training examples
# (the assistant turn is already provided — we don't want an open generation slot)
formatted = tokenizer.apply_chat_template(
    messages,
    tokenize=False,        # return a string, not token IDs
    add_generation_prompt=False,
)
print(formatted)
from transformers import AutoTokenizer
from datasets import Dataset

raw_examples = [
    {"messages": [
        {"role": "user", "content": "What is a REST API?"},
        {"role": "assistant", "content": "A REST API uses HTTP verbs (GET, POST, PUT, DELETE) to manipulate resources identified by URLs, with no client session state stored on the server."},
    ]},
    {"messages": [
        {"role": "system", "content": "Always respond in JSON."},
        {"role": "user", "content": "Name three programming languages."},
        {"role": "assistant", "content": '{"languages": ["Python", "Go", "Rust"]}'},
    ]},
]

def format_example(example):
    return {
        "text": tokenizer.apply_chat_template(
            example["messages"],
            tokenize=False,
            add_generation_prompt=False,
        )

dataset = Dataset.from_list(raw_examples)
formatted_dataset = dataset.map(format_example)

print()
print("=== TOKENIZED LENGTH ===")
toks = tokenizer(formatted_dataset[0]["text"])
print(f"{len(toks['input_ids'])} tokens")

Validating data with pydantic

Data pipelines for fine-tuning are surprisingly brittle: a single malformed example in a 10,000-row JSONL file can crash a training run hours in, or silently corrupt gradients if the tokenizer interprets it differently than expected. The solution is the same as for any data-ingestion pipeline in a backend service: validate at the boundary.

Pydantic is the natural tool here. If you've used FastAPI for request validation, you already know the pattern: define a typed schema, parse every record through it, and fail loudly on malformed input rather than letting errors propagate silently into your model.

From a software-eng POV

Think of training examples as typed DTOs (data transfer objects) that cross the boundary between your data pipeline and the training loop. Pydantic gives you the same safety guarantee as a strongly-typed API contract: if the data doesn't match the schema, you find out at ingestion time — not after a 4-hour GPU run. The same schemas also enforce structured outputs at inference time: define the response schema in pydantic, use model_json_schema() to generate a JSON schema, and pass it to the model as a grammar or tool definition.

python · pydantic schema for training data 
from pydantic import BaseModel, field_validator, model_validator
from typing import Literal
import json

# --- Schema for a single chat training example ---
class Message(BaseModel):
    role: Literal["system", "user", "assistant"]
    content: str

    @field_validator("content")
    @classmethod
    def content_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("Message content must not be blank")
        return v

class ChatExample(BaseModel):
    messages: list[Message]

    @model_validator(mode="after")
    def validate_structure(self) -> "ChatExample":
        roles = [m.role for m in self.messages]
        # Must end with assistant turn
        if roles[-1] != "assistant":
            raise ValueError(f"Last message must be 'assistant', got '{roles[-1]}'")
        # Must have at least one user turn
        if "user" not in roles:
            raise ValueError("Training example must contain at least one user turn")
        return self

# --- Schema for a preference example (DPO) ---
class PreferenceExample(BaseModel):
    prompt: str
    chosen: str
    rejected: str

    @model_validator(mode="after")
    def chosen_differs_from_rejected(self) -> "PreferenceExample":
        if self.chosen.strip() == self.rejected.strip():
            raise ValueError("chosen and rejected must differ")
        return self

# --- Validate a JSONL file at load time ---
def load_chat_dataset(path: str) -> list[ChatExample]:
    valid, errors = [], []
    with open(path) as f:
        for i, line in enumerate(f):
            try:
                record = json.loads(line)
                valid.append(ChatExample.model_validate(record))
            except Exception as e:
                errors.append((i, str(e)))
    if errors:
        print(f"[warn] {len(errors)} invalid examples (first 3):")
        for idx, msg in errors[:3]:
            print(f"  line {idx}: {msg}")
    print(f"Loaded {len(valid)} valid examples.")
    return valid

# --- Usage ---
# examples = load_chat_dataset("train.jsonl")
# Will print a warning for any malformed row instead of crashing silently.

# --- Structured output: generate the JSON schema for inference ---
schema = PreferenceExample.model_json_schema()
print(json.dumps(schema, indent=2))
output
Loaded 2847 valid examples.
[warn] 3 invalid examples (first 3):
  line 12: Last message must be 'assistant', got 'user'
  line 89: Message content must not be blank
  line 203: chosen and rejected must differ

  "title": "PreferenceExample",
  "type": "object",
  "properties": {
    "prompt": {"type": "string"},
    "chosen": {"type": "string"},
    "rejected": {"type": "string"}
  },

}

Synthetic data and distillation

One of the most important shifts in fine-tuning practice since 2023: you don't have to collect all your training data from humans. You can generate it from a stronger model and use it to train a weaker one — this is called distillation.

Analogy · synthetic data = a teacher writing the textbook

Traditional data collection is like assigning homework: you give students (human annotators) problems to solve and collect their answers. Synthetic data is like the teacher writing the textbook directly: GPT-4 or Claude generates the worked examples, and the student model (your smaller model) learns from them. The textbook can be high quality, large-scale, and precisely calibrated to the skills you want to teach — as long as the teacher model knows them well.

The practical workflow: write a diverse set of "seed" prompts covering the tasks you want the model to handle, pass each to a capable model (GPT-4o, Claude 3.5 Sonnet, or even a larger open model like Llama-3-70B) with a careful system prompt specifying the desired output format, validate the outputs with pydantic, filter for quality, and use the passing examples as training data. Touvron et al. showed this works remarkably well for bootstrapping instruction-following capability.

A few caveats apply:

  • Check provider terms of service. Many providers (OpenAI included) prohibit using their API outputs to train competing models. Check before using commercial APIs for distillation.
  • Diversity is critical. If your seed prompts are too similar, the model learns a narrow response pattern and fails to generalize. Aim for broad topic coverage, varying difficulty, and multiple output styles.
  • The teacher's mistakes become the student's curriculum. If the teacher model hallucinates or produces subtly wrong answers in your domain, those errors land in your training set. Always validate outputs against ground truth where possible.

Quality, dedup, and decontamination

The mantra of modern fine-tuning data curation: small and clean beats large and noisy. LIMA demonstrated that 1,000 carefully curated instruction examples could produce a competitive chat model, outperforming models trained on far larger but lower-quality datasets.

Deduplication. Duplicate examples cause overfitting — the model memorizes the duplicated text rather than learning to generalize. Near-duplicate examples are almost as bad. Use MinHash or a simple hash on a normalized form of the text to detect and remove duplicates before training. A 10% dedup pass on a 100k-example dataset is cheap and consistently improves results.

Quality filtering. Common heuristics: remove examples below a minimum token length (stubs that don't teach anything), filter out examples where the response is shorter than the instruction (often a sign of a collapsed or incomplete generation), and use a fast classifier to flag examples that are off-topic or contain known-bad patterns (repetition, profanity where unwanted, formatting artifacts from web scraping).

Decontamination. This one is easy to miss: if your fine-tuning data contains examples from your evaluation benchmarks (MMLU, HumanEval, etc.), your benchmark scores will be artificially inflated — the model has seen the test. This is called data contamination, and it is widespread in published results. Before training, compute n-gram overlap between your training set and your eval sets and remove matches. The threshold is usually: if any 10-gram in the eval example appears in a training example, remove the training example.

How much data do you actually need?

The answer depends on what you're trying to teach, but here are calibrated rules of thumb from practical experience:

Task Typical range Notes
Format / schema enforcement 50–500 examples Models learn format quickly. More examples rarely help; focus on coverage of edge cases.
Persona / tone / style 200–2,000 examples Consistent style across diverse topics. Quality matters more than count.
Narrow domain skill (e.g., SQL generation) 1,000–10,000 examples Cover the space of query patterns. Include edge cases and error handling.
General instruction following (SFT) 10,000–100,000 examples Diverse topic coverage critical. LIMA showed 1k can work if extremely curated.
Preference alignment (DPO) 5,000–50,000 pairs Quality of the preference signal (margin between chosen/rejected) matters more than count.
Continued pretraining (new domain) 1B–100B tokens Domain text, not instruction format. Unstructured corpus is fine here.

The universal advice: start with fewer, higher-quality examples than you think you need. Train a small model (or use fewer steps on the target model), evaluate on a held-out set, then scale up data only if the model is clearly underfitting. Overfitting on a small, curated dataset is a much more common failure mode than underfitting on a large, noisy one.

Recap

  • Three data formats: instruction (flat string, single-turn), chat (messages list with roles, multi-turn), preference (chosen/rejected pairs for DPO). Match the format to your training objective.
  • The chat template is not optional. Use tokenizer.apply_chat_template() always. Always print a decoded example to verify before training. Wrong template → silent failure.
  • Pydantic schemas at ingestion time catch malformed examples before they corrupt training. Treat training data like typed API contracts — validate at the boundary, fail loudly.
  • Synthetic data / distillation from a stronger model can bootstrap a high-quality dataset at low cost. Watch for provider TOS, diversity gaps, and teacher errors.
  • Dedup, quality filter, decontaminate (remove eval-set overlap) before training. These steps are cheap relative to the compute saved by not training on garbage.
  • Small and clean beats large and noisy. Start with fewer examples than you think you need, evaluate, then scale. The most common failure mode is overfitting on a small, noisy dataset — not having too little data.
