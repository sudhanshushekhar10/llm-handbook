Plate 06 · Part II — How LLMs Work

From Weights to Words

A model file is just numbers: billions of floating-point values sitting on disk. This plate traces the path from those static weights all the way to the word that appears on your screen: the forward pass, the probability distribution, and the sampling knobs that tune creativity vs. predictability.

Reading time · 18 min Prereqs · Plate 05 Code · PyTorch · transformers

What "7B parameters" actually counts

When someone says a model has "7 billion parameters," what are they actually counting? A parameter is simply a single floating-point number in a weight matrix (the model's only persistent state between runs). Everything the model "knows" lives in these numbers. During training they are nudged by gradient descent; during inference they are frozen.

The bulk of those 7B live in three places per Transformer block, repeated across N layers:

  • Attention matrices. Each layer has Q, K, V, and output projection matrices. For a model with hidden dimension d_model = 4096 and num_heads = 32, each projection is 4096 × 4096 = ~16.8 M parameters. Four projections per layer × 32 layers = ~2.1 B from attention alone.
  • Feed-forward network (FFN). The FFN typically expands to 4 × d_model then contracts back. For d_model = 4096 that's two matrices of 4096 × 16384 each, roughly 134 M per layer, or ~4.3 B across 32 layers. The FFN is the heavyweight.
  • Embedding + unembedding. A vocabulary of ~32,000 tokens × 4096 dims = ~131 M parameters. These are shared (tied weights) in many models: the same matrix is used to look up input embeddings and to project the final layer back to vocabulary scores.

Add layer norms (tiny), positional encodings (often zero extra params with RoPE), and you land right around 7 B. The "7B" label is therefore a rough proxy for how much GPU RAM you need, how long inference takes, and how much the model "knows", but the FFN is where most of that capacity actually lives.

From a software-eng POV

Think of the model file as a compiled binary: an enormous read-only blob of floats with a well-known layout (layer 0 attention weights at offset X, FFN at offset Y, …). The forward pass is a streaming computation over that blob; nothing is mutated, no heap allocation per request (ignoring the KV cache). Scaling a model means a bigger binary; inference is the "run" step that interprets it.

The forward pass, end to end

You type a prompt. The tokenizer converts it to a sequence of integer IDs. From there, the forward pass is a single, deterministic data-flow computation through the frozen weights:

  1. Token → embedding. Each integer ID is used to look up a row in the embedding matrix: a learned dense vector of length d_model. The sequence becomes a matrix of shape (seq_len, d_model).
  2. N Transformer blocks. Each block runs multi-head self-attention (all tokens talk to each other) followed by a per-token FFN, both wrapped with residual connections and LayerNorm. Information is mixed and refined at each layer. The shape stays (seq_len, d_model) all the way through.
  3. Final norm. A single LayerNorm after the last block stabilizes the magnitudes before the final projection.
  4. Unembedding (the "language model head"). The last token's hidden state (a single vector of length d_model) is multiplied by the unembedding matrix (d_model × vocab_size), yielding a vector of length vocab_size. These raw scores are called logits.

That's it. The entire forward pass is matrix multiplications and element-wise operations. No loops, no branching on content: just a big, parallelisable computation graph that a GPU can run at thousands of tokens per second.

Note a subtle but important point: only the last token's logit vector matters for generation. The earlier tokens produce hidden states used by later tokens via attention (they do real work), but we only read out the top of the stack for the rightmost position. This is why the KV cache is so valuable: for tokens we've already processed, we can skip recomputing their attention keys and values and reuse the cached versions from previous steps.

From logits to a probability distribution

The logit vector has one raw score per vocabulary token: 32,000 numbers for Llama-style models. These scores are not probabilities; they're unbounded and can be negative. We turn them into a proper probability distribution with softmax:

P(tokeni) = exp(zi) / Σj exp(zj)

Softmax does two things: it exponentiates (making all values positive and amplifying differences) and normalises (so values sum to 1). The result is a valid probability distribution, the model's belief about which token comes next, given the entire context so far.

A useful mental model: the logits are scores on an exam; softmax is the grade curve that converts them into percentages. A score 3 points above the mean isn't just "a bit better"; exponentiation makes it e³ ≈ 20× more probable. That sensitivity is exactly what makes temperature scaling so powerful, as we'll see next.

Choosing the next token: greedy, temperature, top-k, top-p

With a probability distribution in hand, the model must sample from it, or skip sampling and just take the peak. This is the decoding strategy, and it has more effect on perceived quality and creativity than almost anything else an engineer controls.

Greedy decoding always picks the argmax: the single highest-probability token. Fast, deterministic, and surprisingly bad for open-ended generation: it leads the model into repetition loops and "safe" but boring outputs because it never explores lower-probability options that might open up richer continuations.

Temperature is a scalar applied to logits before softmax: P = softmax(z / T). At T = 1.0 (the default), you get the raw distribution. At T < 1.0 (e.g. 0.3), differences are amplified; the distribution sharpens and the model becomes more confident and predictable. At T > 1.0 (e.g. 1.5), differences are compressed; the distribution flattens and low-probability tokens get a bigger share. Higher temperature = more varied, more surprising, sometimes more creative, sometimes more nonsensical.

Top-k is a hard truncation: zero out all but the top k logits before softmax. This prevents any truly bizarre low-probability tokens from sneaking through, regardless of temperature. A typical value is k = 50. The downside: the right k varies wildly by position. Sometimes the distribution is genuinely flat across 1000 plausible tokens; sometimes only 3 are reasonable.

Top-p (nucleus sampling) is adaptive: keep the smallest set of tokens whose cumulative probability exceeds threshold p. With p = 0.9, on a "flat" step you keep many tokens; on a "certain" step you might keep just 2. Nucleus sampling self-tunes to the model's current uncertainty, and is why "top-p = 0.9, temperature = 0.8" is the default recipe for most chat models.

Repetition penalty divides logits of recently seen tokens by a penalty scalar (> 1.0), reducing their probability. It's a band-aid for repetition loops, not a cure, but it works well enough to be standard.

Analogy · weighted dice

The model computes a probability distribution. Picking the next token is literally rolling a loaded die with 32,000 faces. Temperature adjusts how loaded it is: cold (low T) makes the die almost always land on the same face; hot (high T) makes every face nearly equally likely. Top-k and top-p tape over the faces that are too unlikely before you roll: they constrain the sample space without changing which faces remain.

Gotcha · temperature 0 ≈ greedy, but not truly deterministic

Setting temperature=0 is supposed to give you determinism. In practice, most frameworks implement it as greedy (argmax), which is deterministic for a given model on a given device. But floating-point non-associativity across GPU kernels, quantization, and non-deterministic CUDA ops mean that the same prompt on different hardware (or even different runs on the same hardware with CUDA_LAUNCH_BLOCKING=1 not set) can yield different results. True reproducibility requires a fixed seed and deterministic ops and identical hardware/software. In production, treat LLM output as non-deterministic by default.

Here's a from-scratch illustration of sampling on a small logit vector:

python · softmax + sampling from scratch 
import torch
import torch.nn.functional as F

# Tiny example: 8 vocab tokens with raw logit scores
logits = torch.tensor([1.2, 0.8, 3.1, -0.5, 2.4, 0.1, -1.0, 1.8])
vocab  = ["the", "a", "cat", "dog", "sat", "ran", "over", "mat"]

def sample_next_token(logits, temperature=1.0, top_p=0.9, top_k=0):
    # 1. Temperature scaling
    logits = logits / max(temperature, 1e-6)

    # 2. Optional top-k hard truncation
    if top_k > 0:
        kth_val = torch.topk(logits, top_k).values[-1]
        logits = logits.masked_fill(logits < kth_val, float("-inf"))

    # 3. Softmax → probability distribution
    probs = F.softmax(logits, dim=-1)

    # 4. Top-p nucleus: keep smallest set summing to >= top_p
    if top_p < 1.0:
        sorted_probs, sorted_idx = torch.sort(probs, descending=True)
        cumulative = torch.cumsum(sorted_probs, dim=-1)
        # remove tokens once cumulative prob exceeds top_p
        remove = cumulative - sorted_probs > top_p
        sorted_probs[remove] = 0.0
        # renormalise
        sorted_probs /= sorted_probs.sum()
        # sample from the truncated distribution
        chosen = torch.multinomial(sorted_probs, 1)
        return sorted_idx[chosen].item()

    return torch.multinomial(probs, 1).item()

# Greedy (effectively temperature=0)
greedy = logits.argmax().item()
print(f"Greedy:            '{vocab[greedy]}'")  # always the argmax

# Temperature 0.5 — sharper, more predictable
torch.manual_seed(42)
idx05 = sample_next_token(logits, temperature=0.5, top_p=0.9)
print(f"T=0.5, top-p=0.9:  '{vocab[idx05]}'")

# Temperature 1.5 — flatter, more varied
torch.manual_seed(42)
idx15 = sample_next_token(logits, temperature=1.5, top_p=0.9)
print(f"T=1.5, top-p=0.9:  '{vocab[idx15]}'")
output
Greedy:            'cat'       # token index 2, highest raw logit (3.1)
T=0.5, top-p=0.9:  'cat'       # still usually 'cat' — cold distribution is peaky
T=1.5, top-p=0.9:  'sat'       # temperature flattened the distribution; 'sat' won the roll

The autoregressive loop

The model doesn't generate the full response in one shot. It generates one token, then feeds that token back as part of the new input, then generates the next token, and so on. This is the autoregressive loop: the fundamental reason LLM generation is sequential and why latency scales with output length.

Concretely: at step t, the input sequence is [prompt tokens + token_1 + … + token_{t-1}]. The model runs the full forward pass and reads the logits at position t. We sample token_t, append it to the context, and repeat. The context window grows by one token per step until we hit a stop token (<|eos|>) or the maximum length.

The KV cache is what makes this tractable. Without it, each step would recompute attention over the entire sequence: O(n²) work per step, O(n³) total. With the KV cache, we store the key and value vectors from previous steps and only compute attention for the new token, reducing each step to roughly O(n) (a table lookup into the cached keys and values).

From a software-eng POV · it's a REPL with state

The autoregressive loop is structurally identical to a REPL (Read-Eval-Print Loop) where the "eval" step is the forward pass, the "print" appends one token to the buffer, and the "read" re-ingests the whole buffer. The KV cache is the memoisation layer that avoids recomputing intermediate results for the prefix you've already seen, exactly like a dynamic programming table or an HTTP response cache keyed on the request prefix.

Here's the full generation loop using Hugging Face transformers, with the Local/Cloud tab showing how to vary sampling parameters:

python · model.generate with sampling 
# pip install torch transformers
# Runs on Apple Silicon (MPS) or CPU — uses a small model for speed.
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

device = "mps" if torch.backends.mps.is_available() else "cpu"
model_id = "HuggingFaceTB/SmolLM2-135M-Instruct"
tok   = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id, torch_dtype=torch.float16).to(device)

prompt = "The three most important things a software engineer should know about LLMs are"
inputs = tok(prompt, return_tensors="pt").to(device)

# ---- Greedy (deterministic) ----
greedy = model.generate(**inputs, max_new_tokens=40, do_sample=False)
print("=== Greedy ===")
print(tok.decode(greedy[0], skip_special_tokens=True))

# ---- Temperature + top-p (creative) ----
sampled = model.generate(
    **inputs, max_new_tokens=40, do_sample=True,
    temperature=0.8, top_p=0.9, repetition_penalty=1.1,
)
print("\n=== Sampled (T=0.8, top-p=0.9) ===")
print(tok.decode(sampled[0], skip_special_tokens=True))

Why outputs vary, and how to get determinism

The most common question engineers ask when first integrating an LLM API: "Why does this endpoint return different answers every time I call it with the same prompt?" The answer is now obvious: sampling is probabilistic. The model computes the same deterministic forward pass and lands on the same distribution, but then rolls the dice. Different rolls, different tokens, different outputs.

This variance is a feature for creative tasks and a bug for structured extraction. Here's the toolkit:

GoalSettingTrade-off
Determinism / reproducibilitytemperature=0 (greedy); set seedBoring, repetitive; hardware-dependent in practice
Precise / factual outputtemperature=0.1–0.3, top_p=0.95Low variance, less creative
Balanced qualitytemperature=0.7–0.9, top_p=0.9Default for most chat APIs
Creative / brainstormingtemperature=1.0–1.3, top_p=0.95Varied, occasionally incoherent
Structured JSON outputConstrained decoding / grammar samplingGuaranteed schema; slight quality cost

The autoregressive nature also means early randomness compounds: a surprising first word steers the model into a different region of probability space, producing a chain of choices that diverges completely from another run. This is why two runs at temperature=0.8 can produce answers that look like they came from different models.

For most production systems the practical answer is: accept stochasticity, test with a distribution of outputs, and use structured output constraints (structured output) when you need a schema. Reserve temperature 0 for cases where you genuinely need the same string every time, and even then, document the hardware dependency.

Recap

  • A "7B model" has ~7 B floating-point parameters split mainly between attention matrices and FFN matrices across N layers, plus the embedding/unembedding tables.
  • The forward pass is: token IDs → embedding lookup → N Transformer blocks → final norm → unembedding → logits (one score per vocab token).
  • Softmax converts unbounded logits into a probability distribution; the sensitivity to score differences is exponential.
  • Temperature sharpens (low) or flattens (high) the distribution before softmax. Top-k hard-truncates; top-p (nucleus) adapts the cutoff to the model's current uncertainty.
  • The autoregressive loop generates one token at a time, feeds it back in, and uses the KV cache to avoid recomputing all prior attention states.
  • Output variance is a direct consequence of probabilistic sampling, not a bug. Temperature 0 gives near-determinism but depends on hardware.
