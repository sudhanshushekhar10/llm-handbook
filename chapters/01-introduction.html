The Mental Model

Before you touch a line of code, you need a sharp mental model of what a language model actually is — not the marketing pitch, but the engineering reality. It's a function. A big one. And that reframe changes everything.

Reading time · 12 min Prereqs · None — start here Code · transformers

Software 1.0 vs 2.0: code you write vs weights you grow

In 2017 Andrej Karpathy coined the phrase Software 2.0 for a shift that was already happening underfoot. In the old world — Software 1.0 — a programmer writes explicit rules: if sentiment == "positive" then score += 1. The program is a text file you can read, diff, and reason about. You author it.

Software 2.0 inverts this. You don't write the rules; you write the objective and collect examples. Then you run an optimization process — training — that searches for a set of numbers (the parameters, or weights) that minimize your objective across those examples. The rules are never written explicitly. They are inferred from data and encoded as billions of floating-point numbers inside a file.

Analogy · grown, not written

Think of the difference between hand-crafting a regex to parse addresses (Software 1.0) versus training a model on a million labelled address strings and letting it figure out the rules (Software 2.0). In the first case your logic lives in your source code. In the second it lives in a weight matrix — a set of numbers that implicitly encode a pattern you never had to articulate. The surprising thing is that this approach, scaled up, generalizes far beyond what any hand-crafted ruleset could.

This isn't just a philosophical difference. It changes what "debugging" means, what "deploying" means, and what "versioning" means. Instead of shipping a new release of your service, you might ship a new fine-tuned model file. Instead of adding an if branch to handle a new case, you add training examples. The mental models you built in Software 1.0 still apply — but the artefacts are different.

A model is just a (very large) function

Strip away the hype and a language model is this:

f(tokens) → probability distribution over the next token

You pass in a sequence of tokens (integers representing chunks of text — more on this in Plate 04). The function runs some arithmetic and returns a vector of scores, one per word in the model's vocabulary. After a softmax, those scores become probabilities: given everything I've seen so far, here's how likely each possible next word is.

Those raw scores are called logits — a term borrowed from statistics that just means "unnormalized log-probabilities." The function that produces them is a Transformer — a specific neural-network architecture — and its weights are the learned parameters: typically billions of them, each a 16- or 32-bit float.

From a software-eng POV

A model is a pure function with no side effects. Give it the same input, get the same logits back every time (during inference with deterministic settings). The "intelligence" isn't hidden state or a database query — it's entirely encoded in the weight matrices that parameterize the function. The model file is your build artifact: immutable, versioned, deployable.

The function is autoregressive: to generate a full sentence, you call it once to get the first token, append that token to the input, call it again, and repeat. Each call is one forward pass — thousands of matrix multiplications, executed in parallel on a GPU. The "text" you see is just the model calling itself over and over, each time looking at everything it has produced so far, plus your original prompt.

Train, infer, fine-tune — the three verbs

The entire LLM world is organized around three operations. Get these straight and the rest of this guide clicks into place.

VerbWhat happensWho does itAnalogy
Pretraining Run gradient descent on trillions of tokens of text; the weights are adjusted to predict each next token. This is where the model "learns" language, facts, code, and reasoning patterns. A handful of labs (OpenAI, Meta, Google, Mistral…) — costs millions of dollars of GPU time Compiling the OS from scratch
Inference Call the frozen function on a new input; read off the logits; sample a token; repeat. No weights change. Anyone with the model file and a GPU (or an API key) Running a compiled binary
Fine-tuning Continue training — but on a smaller, curated dataset, to steer the model toward a specific task or style. Weights change again, but starting from a pretrained checkpoint. Teams customizing a base model for their domain or safety requirements Patching or extending a library

Pretraining is the expensive, once-per-model-generation step. Everything else in this guide is downstream of it. When someone says "GPT-4" or "Llama 3," they mean a specific set of frozen weights that resulted from one enormous pretraining run. Inference is the everyday operation — it's what runs when you hit the API. Fine-tuning sits between the two: cheaper than pretraining by many orders of magnitude, but it does change the weights, so you end up with a new model file.

Sampling is the step that makes inference feel non-deterministic. After you have a probability distribution over the next token, you could take the highest-probability token every time (greedy decoding) — but that produces repetitive, robotic text. Instead you sample: draw from the distribution, biased by a temperature knob. Turn temperature up and you get more creative (more randomness); turn it down to zero and you're back to greedy. Same weights, different sampling strategy, dramatically different outputs.

Where models actually live: files, frameworks, GPUs

A language model is, at its core, a file — or rather a collection of files. The weights are usually serialized as tensors (multi-dimensional arrays of floats) in a format like safetensors or the older pytorch_model.bin. A 7-billion-parameter model in 16-bit precision occupies roughly 14 GB on disk and in VRAM (GPU memory). A 70B model: around 140 GB.

The code that runs those weights is almost universally PyTorch — a Python library from Meta that treats tensors as first-class objects, runs operations on GPUs via CUDA (NVIDIA's GPU programming layer), and tracks gradients for training. You don't typically write raw PyTorch for inference; you use Hugging Face's transformers library, which wraps models in a consistent API and handles the tokenization, batching, and sampling for you.

On Apple Silicon machines, CUDA is replaced by MPS (Metal Performance Shaders) — Apple's GPU compute layer. Smaller models (up to ~13B parameters) run comfortably on a MacBook Pro M-series, which is a surprisingly capable inference machine for development and experimentation.

python · hello, model 
# pip install transformers torch
from transformers import pipeline

# A small (135M param) instruction-tuned model — fits in ~300 MB of VRAM
gen = pipeline(
    "text-generation",
    model="Qwen/Qwen2.5-0.5B-Instruct",
    device="mps",          # use "cuda" on NVIDIA, "cpu" as fallback
    max_new_tokens=40,
)

result = gen("Explain what a language model is in one sentence:")
print(result[0]["generated_text"])
output
Explain what a language model is in one sentence:
A language model is a statistical system trained to predict the next
token in a sequence, which allows it to generate coherent text.
# ~2 s on M2 Pro; no API key, no network after the first model download

Notice what just happened: you downloaded ~300 MB of weight files once (cached under ~/.cache/huggingface), loaded them into memory, and called a function. No server, no API key, no internet after the initial pull. The model is a file; inference is a function call. That's the whole picture.

It's a predictor, not a database

The most important thing to internalize — and the source of the most frustrating bugs — is that a language model has no memory and no source of truth. It is a predictor: it returns whatever sequence of tokens is most probable given its training data. It has no index to look up, no database to query, no verified facts to serve.

This is why hallucination is not a bug that will be fixed in the next version. It is a structural property of the architecture. The model learned that after "The capital of France is" the token "Paris" is very likely — but it learned this by observing the statistical pattern in text, not by consulting a gazetteer. For common, frequently-discussed facts this correlation is reliable. For obscure facts, recent events, or precise figures, the model will confidently produce the most plausible-sounding continuation regardless of whether it is true.

Gotcha · predictor, not database

A model that says "the function str.format() takes a locale argument" sounds authoritative and is completely wrong. It produced the most statistically plausible API description — but plausibility is not accuracy. Treat every factual claim as a draft that needs verification. This is not a temporary limitation; it follows directly from what a model is. Solutions (RAG, tool use, citations) work around it, but do not eliminate it.

This distinction maps cleanly onto something engineers already know: a model is closer to a compiler learned from code examples than to a database. A compiler that learned patterns might produce syntactically valid output that is semantically wrong. That's exactly what happens here — the output is grammatically and stylistically fluent, but the facts it expresses are unchecked.

Recap

  • Software 2.0: a model's "logic" is not written — it is inferred from data via training and encoded as billions of floating-point weights.
  • A language model is a pure function: tokens in → probability distribution over the next token out. Same input, same logits. No side effects.
  • Autoregressive generation means calling that function token-by-token, feeding each new output back as input, until you have a full response.
  • The three key verbs: pretraining (grow the weights from scratch), inference (call the frozen function), fine-tuning (continue training to adapt).
  • A model is a file of numbers plus a little PyTorch code to run it. Hugging Face's transformers library provides the standard API; GPUs (CUDA or MPS) provide the compute.
  • A model is a predictor, not a database. Hallucination is structural, not a bug — it follows from predicting plausible continuations rather than retrieving verified facts.
